# GB16 — при добавлении условия колонка не должна быть выбрана заранее

Прочитать перед началом: `Components/Grid/Filter/ClayFilterGroup.razor.cs` — `AddExpression`;
`Components/Grid/Filter/ClayFilterExpression.razor.cs` — `OnParametersSet`, `RefreshDescriptor`,
`OnColumnChanged`, `_descriptor`, `_availableOperators`, поле `Node.IsNew` и `OnAfterRender`;
`Components/Grid/Filter/ClayFilterExpression.razor` — `MudSelect` колонки (пустой `Value` →
плейсхолдер), `MudSelect` оператора (`Disabled="@(_descriptor is null)"`), блок значения
`@if (_descriptor is not null)`; `Components/Grid/Filter/ClayFilterDialog.razor.cs` — `Apply`
и как отбрасываются неполные условия; `Components/Grid/Filter/ClayCompositeSqlBuilder.cs` —
`BuildLeaf` (как ведёт себя лист без колонки при построении SQL);
`Components/Grid/Filter/ClayFilterGroupNode.cs` / `ColumnFilter` — поля `Column`, `IsNew`.

## Дефект

Кнопка «Добавить условие» создаёт лист с уже выбранной первой колонкой:

```csharp
private async Task AddExpression()
{
    var firstCol = Columns.FirstOrDefault();
    var newLeaf  = new ColumnFilter
    {
        Column    = firstCol?.SqlName ?? "",
        ParamName = "",
        Operator  = firstCol?.Type.DefaultOperator ?? ColumnFilterOperator.Contains,
        Source    = ClayFilterSource.CompositeDialog,
    };
    Node.Nodes.Add(newLeaf);
    await NotifyChanged();
}
```

До GB15 «первая колонка» была первой по порядку регистрации и на глаз выглядела произвольной.
После GB15 список отсортирован по алфавиту, поэтому в поле стабильно подставляется колонка,
которая сортируется первой (условно «А…»), — и это читается как «система уже что-то выбрала за
меня». Пользователь хочет выбрать колонку сам: новое условие должно открываться с пустым полем
колонки.

GB15 не сломал ничего — он лишь сделал давно заложенную предвыборку заметной. Правка не в
сортировке, а в предвыборке.

Компонент выражения к пустой колонке уже готов: `RefreshDescriptor("")` даёт `_descriptor = null`,
при `null` оператор-селект `Disabled`, блок значения (`@if (_descriptor is not null)`) не
рендерится, `MudSelect` колонки с пустым `Value` показывает `Label` как плейсхолдер. То есть
пустой стартовый лист — валидное «ещё не выбрано» состояние, специально ничего рисовать не надо.

Нужно убедиться в двух местах, что пустой лист не ломает применение фильтра:
1. `Apply` в `ClayFilterDialog` — незавершённые листья (без колонки) должны отбрасываться, а не
   уходить в SQL;
2. `ClayCompositeSqlBuilder.BuildLeaf` — на всякий случай не должен падать/генерировать мусор на
   листе с пустой `Column`.

## Изменить/создать

### 1. `ClayFilterGroup.razor.cs`, `AddExpression` — не подставлять колонку

```csharp
/// <summary>
/// Добавляет новое пустое листовое условие (Source=CompositeDialog). Колонка НЕ
/// выбирается заранее — пользователь выбирает её сам (пустое поле = плейсхолдер).
/// Оператор и значение появятся после выбора колонки (OnColumnChanged выставит
/// оператор по умолчанию для типа).
/// </summary>
private async Task AddExpression()
{
    var newLeaf = new ColumnFilter
    {
        Column    = "",
        ParamName = "",
        Operator  = ColumnFilterOperator.Contains,   // плейсхолдер; переопределится при выборе колонки
        Source    = ClayFilterSource.CompositeDialog,
        IsNew     = true,   // сохраняем автофокус после первого рендера — если поле было завязано на это
    };
    Node.Nodes.Add(newLeaf);
    await NotifyChanged();
}
```

`Columns.FirstOrDefault()` и производные (`firstCol?.Type.DefaultOperator`) убрать —
предвыборки больше нет. Если `IsNew` использовался для автофокуса поля ЗНАЧЕНИЯ (см.
`OnAfterRender` в `ClayFilterExpression`), учти: значения ещё нет (колонка не выбрана), фокус
логичнее навести на селект колонки. Если сделать фокус на колонке несложно (например, через
`AutoFocus`/`@ref` у `MudSelect`) — наведи; если это тянет заметную возню — оставь без
автофокуса и отметь в отчёте, автофокус здесь не критичен.

### 2. `ClayFilterExpression.razor` — плейсхолдер в селекте колонки

Убедись, что у `MudSelect` колонки задан `Placeholder` (или `Label` работает как таковой при
пустом `Value`), чтобы пустое поле читалось как «выберите колонку», а не как пустая строка.
Если плейсхолдера нет — добавь `Placeholder="@ClayFilterStrings.FieldPlaceholder"` и заведи
строку в `ClayFilterStrings` (напр. «Выберите колонку»). Визуальных инлайнов не добавлять
(StyleGuard) — только текст.

### 3. Применение фильтра — отбросить незавершённый лист

Проверь `Apply` в `ClayFilterDialog.razor.cs`: лист с пустой `Column` (пользователь нажал
«Добавить условие», но колонку не выбрал) не должен попадать в результат и в SQL. Если такая
отбраковка уже есть (по `HasValue`/непустой колонке) — ничего не добавляй. Если нет — при сборке
результата пропускать листья с `string.IsNullOrEmpty(Column)`. Пустые группы, оставшиеся после
отбраковки, — по существующей логике (если она их чистит — хорошо; специально расширять не надо).

Аналогично убедись, что `ClayCompositeSqlBuilder.BuildLeaf` на листе с пустой `Column` либо не
вызывается (отбракован раньше), либо безопасно возвращает пустое условие, а не падает.

## Не делай

- Не откатывай GB15 (сортировку колонок) — она корректна, дефект не в ней.
- Не подставляй «первую» колонку никаким иным способом (по порядку грида, по частоте и т.п.) —
  требование заказчика: пользователь выбирает колонку сам.
- Не блокируй кнопку «Применить», пока есть незавершённый лист, и не показывай ошибку — просто
  молча отбрасывай листья без колонки при применении (пользователь мог добавить лишнюю строку
  и передумать).
- Не трогай листья `Source=ColumnDialog` и `ValueFilter` — они не редактируются в этом
  компоненте, к ним предвыборка отношения не имеет.
- Не меняй `OnColumnChanged` (он правильно выставляет оператор по умолчанию и сбрасывает
  значение при первом выборе колонки).
- Не трогай второй баг из этого прогона (двойной скролл) — это отдельный шаг GB17.

## Проверка (ручная)

- `Kesco.App.Web.Inventory`, `?id=140`: «Настраиваемый фильтр» → «Добавить условие» → поле
  колонки ПУСТОЕ (плейсхолдер «Выберите колонку»), селект оператора неактивен, поля значения нет;
- выбрать колонку → появился оператор (значение по умолчанию для типа) и редактор значения;
- добавить два условия, у обоих поля пустые, заполнить одно, второе оставить пустым → «Применить»
  → фильтр применился по заполненному, пустое молча отброшено, грид отфильтрован корректно;
- добавить условие, не выбрать колонку, «Применить» → фильтр пуст (или без этого условия),
  ошибок нет;
- вложенная группа: «Добавить условие» внутри неё → тоже пустое поле;
- ранее сохранённый фильтр (с выбранными колонками) открывается с правильно выбранными
  колонками — пустое стартовое состояние только у НОВЫХ условий;
- статический режим (`/medical-tests`): «Добавить условие» → поле колонки пустое, поведение
  идентично;
- `dotnet build` + `dotnet test` — зелёные.
