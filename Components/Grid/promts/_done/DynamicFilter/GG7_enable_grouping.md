# GG7 — включение группировки в динамическом режиме

**Это шаг включения фичи. Если GG1–GG6 не выполнены — ОСТАНОВИСЬ.** До этого момента весь
написанный код лежал мёртвым: `Groupable = false`, кнопки «Группировать» нет. Здесь всё
оживает, и здесь же вылезет всё, что было сделано неаккуратно.

Прочитать перед началом (обязательно, до написания кода):

- `Components/Grid/ClayGrid.Dynamic.cs` — главный цикл регистрации `ClayColumnMeta` в
  `InitDynamicMode` (там `Groupable = false` после GF14), блок регистрации фильтр-онли
  колонок (Тип 6/11), `RestoreDynamicState`, `ApplySavedGroups`.
- `promts/GF14_disable_dynamic_grouping.md` — что именно там было отключено. **Этот шаг
  частично откатывает GF14.**
- `Components/Grid/ClayGrid.razor` — кнопка «Группировать»
  (`@if (_columnBySqlName.Values.Any(c => c.Groupable))`), лоток, `Hidden="@IsGrouped(sqlName)"`.
- `Components/Grid/ClayGrid.Grouping.cs` — `AddGroupColumn`, `RemoveGroupColumn`, `_groupColumns`.
- `Components/Grid/ClayGrid.DragDrop.cs` — `OnTrayDrop` (как колонка попадает в `_groupColumns`).
- `Components/Grid/Dynamic/GridStateSerializer.cs` — `SerializeGroups` / `DeserializeGroups`.
- `Components/Grid/ClayGrid.Dynamic.Grouping.cs` — весь код GG2–GG6.
- **`GG0_README_dynamic_grouping.md`, раздел «Модель доверия» — прочитай обязательно.**

## Задача

Три изменения:

1. вернуть `Groupable = true` колонкам вывода — кнопка «Группировать» и лоток появятся сами;
2. вернуть восстановление группировки из `grp+gridId`;
3. сбрасывать раскрытые группы при смене состава группировки — иначе `_dynamicExpandedGroups`
   накопит ключи от старой группировки.

## Изменить/создать

**1.** `ClayGrid.Dynamic.cs`, главный цикл регистрации меты — вернуть `Groupable = true`:

```csharp
var meta = new ClayColumnMeta
{
    ColumnId    = col.ColumnId,
    SqlName     = col.Column,
    DisplayName = col.Header ?? col.Column,
    SortName    = col.Column,
    Groupable   = true,
    Filterable  = col.Type != (int)ClayColumnKind.List,
    Type        = desc,
};
```

Комментарий про «группировка не реализована», добавленный в GF14, удалить — он больше не верен.

Блок регистрации фильтр-онли колонок (Тип 6/11) НЕ трогать: там `Groupable = false` был
изначально, до GF14, и должен таким остаться — этих колонок нет в выводе, группировать по ним
нечего.

**2.** `ClayGrid.Dynamic.cs`, `RestoreDynamicState` — вернуть вызов:

```csharp
        // Группировка
        var grpName = p(opt.GroupingParamPrefix);
        if (saved.TryGetValue(grpName, out var grpVal))
            ApplySavedGroups(grpVal);
```

`ApplySavedGroups` уже написан и уже делает главное:

```csharp
foreach (var sqlName in groups)
{
    if (_columnBySqlName.ContainsKey(sqlName))     // ← БЕЛЫЙ СПИСОК, НЕ УДАЛЯТЬ
        _groupColumns.Add(sqlName);
}
```

**Проверка `_columnBySqlName.ContainsKey(sqlName)` — не формальность, а защита от SQL-инъекции.**
`grp140` — строка из таблицы `vwНастройки`, а имена группировочных колонок подставляются в
`GROUP BY {expr}` и `{exprs[i]} = @dk{i}` текстом, без параметризации (см. GG2,
`ClayGroupingEngine.BuildGroupAggregateSql`). Уберёшь фильтр — получишь исполнение произвольного
SQL из таблицы настроек. Убедись, что проверка на месте, и не трогай её.

Убедись также, что `ApplySavedGroups` вызывается ПОСЛЕ регистрации колонок (в `InitDynamicMode`
цикл регистрации идёт до `RestoreDynamicState`) — иначе `_columnBySqlName` пуст и белый список
отбросит всё.

**3.** `ClayGrid.Dynamic.Grouping.cs` — сброс раскрытых групп при смене группировки. Добавь
метод и зови его оттуда, где меняется состав `_groupColumns`:

```csharp
    /// <summary>
    /// Сбрасывает раскрытые группы. Вызывается при изменении состава/порядка группировки:
    /// ключи групп (FullKey) строятся из значений группировочных колонок, при другой
    /// группировке они бессмысленны и будут молча висеть в наборе.
    /// </summary>
    private void ResetDynamicExpandedGroups() => _dynamicExpandedGroups.Clear();
```

`ClayGrid.Grouping.cs`, `AddGroupColumn` и `RemoveGroupColumn`:

```csharp
    private async Task AddGroupColumn(string column)
    {
        if (_groupColumns.Contains(column))
            return;
        _groupColumns.Add(column);
        if (Dynamic) ResetDynamicExpandedGroups();
        _pageNumber = 1;
        await NotifyQueryChanged();
    }

    private async Task RemoveGroupColumn(string column)
    {
        _groupColumns.Remove(column);
        if (Dynamic) ResetDynamicExpandedGroups();
        _pageNumber = 1;
        await NotifyQueryChanged();
    }
```

Найди в `ClayGrid.DragDrop.cs` (`OnTrayDrop`) все места, где `_groupColumns` меняется НАПРЯМУЮ,
минуя эти два метода (перестановка чипов внутри лотка). Там тоже нужен сброс. **Прочитай файл
и найди их сам** — если таких мест нет, ничего не добавляй и скажи об этом.

## Не делай

Не меняй формат `grp+gridId` (`col1,col2` через запятую) — он уже в БД. Не убирай белый список
в `ApplySavedGroups`. Не ставь `Groupable = true` фильтр-онли колонкам (Тип 6/11).
Не сохраняй `_dynamicExpandedGroups` в `vwНастройки` (см. GG3). Не трогай `SaveDynamicState` —
`grp+gridId` там уже сохраняется. Не трогай статический режим. Чекбоксы групп в этом шаге не
делаются — это GG8.

## Проверка (ручная, полная)

Временных хаков из GG2–GG6 в коде остаться не должно — **проверь, что `_groupColumns.Add("КодТипа")`
из `InitDynamicMode` удалён**.

Одноуровневая, `?id=140&CLID=9`:

- в тулбаре появилась кнопка «Группировать»; нажать → открылся лоток с подсказкой
  «Перетащите заголовок колонки сюда для группировки»;
- перетащить заголовок «Тип исследования» в лоток → грид сгруппировался, появились заголовки
  групп с НАИМЕНОВАНИЯМИ типов и счётчиками, колонка `Тип исследования` скрылась из грида;
- в меню (⋮) заголовка колонки при `ColumnMenuMode="Always"` появился пункт «Группировать» —
  работает так же;
- шеврон раскрывает/сворачивает группу; кнопка на чипе разворачивает/сворачивает все;
- крестик на чипе → грид вернулся в плоский режим, колонка вернулась;
- «Всего» и число страниц пересчитываются, листание работает.

Двухуровневая (главная проверка, раньше была недоступна):

- перетащить в лоток вторую колонку → двухуровневое дерево: внешние группы, внутри —
  подгруппы с отступом (`padding-left: Depth * 16px`), внутри — детали;
- счётчик внешней группы = сумма счётчиков её подгрупп (`ComputeParentCounts`);
- раскрыть внешнюю, потом внутреннюю → детали видны; свернуть внешнюю → внутренние скрылись;
- кнопка уровня 1 (второй чип) → раскрылись все подгруппы И их родители (каскад);
- поменять чипы местами перетаскиванием → дерево пересобралось, раскрытые группы сброшены,
  зависших ключей нет;
- **перетащить в лоток ТРЕТЬЮ колонку → третий уровень НЕ работает** (`BuildAggregates` читает
  только K0/K1, см. GG0 «Ограничения»). Это ожидаемое существующее ограничение, оно же в
  статическом режиме. Зафиксируй фактическое поведение в отчёте и НЕ чини.

Персистенция:

- `?id=140&CLID=7` → сгруппировать по «Тип исследования», F5 → группировка восстановилась,
  группы свёрнуты (раскрытость — состояние сессии);
- `?id=140&CLID=9` → группировки нет (чужое состояние не влияет);
- вручную вписать в `vwНастройки` `grp140` = `НетТакойКолонки` для `CLID=7` → грид плоский,
  ошибок нет, инъекции нет (белый список);
- вручную вписать `grp140` = `КодТипа); DROP TABLE Типы--` → грид плоский, таблица `Типы`
  на месте. **Эту проверку выполнить обязательно.**

Совместная работа:

- группировка + фильтр → агрегат ушёл с `WHERE` фильтра, счётчики групп учитывают фильтр;
- группировка + поиск → то же, ошибки `@search` нет (GF16);
- группировка + сортировка по группировочной колонке → порядок групп сменился
  (`BuildGroupAggregateSql`, `ORDER BY`);
- группировка + сортировка по НЕгруппировочной колонке → порядок деталей внутри групп
  сменился (`BuildDetailOrder`);
- группировка + скрыть колонку через «Настройку колонок» → заголовок группы не уехал в
  скрытую колонку (`GroupRowHostKey` пропускает `_hiddenSqlNames`);
- группировка + карандаш/корзина на строке детализации → работают.

Статический режим (`MedicalTests.razor`): группировка, раскрытие, чипы, автопереход —
всё как раньше.
