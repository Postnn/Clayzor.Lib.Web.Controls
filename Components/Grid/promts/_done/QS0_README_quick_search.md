# QS0 — Быстрый поиск в динамическом ClayGrid (индекс)

> **Читать первым. Шаги выполнять строго по порядку, каждый шаг — отдельный файл.**
> Стиль исполнения — по корневому `AGENTS.md`: **Think Before Coding → Simplicity First → Surgical Changes → Goal-Driven Execution.**

## Глобальные правила (действуют на все шаги)

- Весь пользовательский текст и Razor-разметка — **на русском**.
- Каждый новый public/protected класс, метод, свойство, поле — с `/// <summary>`.
- Каждая новая SQL-константа — `///` XML-doc + `--` комментарий на **каждую** колонку.
- **SQL Server 2008 R2** — только доступный в этой версии синтаксис. Никаких `OFFSET/FETCH`, `TRY_CONVERT`, `STRING_SPLIT`, `CONCAT_WS`, `IIF`.
- **Запрещено** подставлять значения в SQL-строку. Все параметры — через Dapper (`@param`).
- При `Db.ExecuteAsync()` с сырым SQL обязательно `commandType: CommandType.Text`.
- Имена колонок БД — точные русские, не переводить. Новое имя колонки определить **ровно один раз** в `ColumnNames.cs`.
- Git (add/commit/push) — **только по прямому указанию**. `AGENTS.md` и `docs/` **не** обновлять.
- Если требования неоднозначны — **задать уточняющий вопрос через question tool**, озвучив свои предположения. Не решать молча.
- После каждого шага — прогнать `dotnet build Clayzor.sln` и `dotnet test tests\Clayzor.Lib.Web.Controls.Tests`. Не переходить к следующему шагу на красной сборке.

## Задача

В **динамическом** режиме `ClayGrid` появляется быстрый поиск по значению, введённому в поле поиска на панели управления грида.

1. В `ColumnsTable` (`dbo.ClayGridColumns`) добавлена колонка **`УчаствуетВБыстромПоиске tinyint`**. Значения: `0` — не участвует, `1` — участвует.
2. Значение по умолчанию задаёт админ при создании запроса динамического грида.
3. Пользователь может менять участие колонок в поиске **в диалоге настройки колонок**.
4. Пользовательский выбор сохраняется в `UserParamsTable` (`dbo.ClayGridUserParams`). В параметр сохраняются **только названия колонок, участвующих в поиске**. Константа-префикс параметра — **`QuickSearchParamPrefix`**.
5. Участвовать в быстром поиске могут **только колонки тех типов, которые имеют явное представление в источнике данных**.
6. Поле поиска **скрыто**, если:
   - все `УчаствуетВБыстромПоиске` в колонках запроса равны `0`/`NULL`, **или**
   - колонки `УчаствуетВБыстромПоиске` **физически нет** в `ColumnsTable` — тогда функционала быстрого поиска в гриде нет вообще.

## Проверенные факты по схеме (взяты из `scripts/dynamic-grid/schema.sql`, не выдумывать заново)

```
dbo.ClayGridColumns:   КодКолонки int PK, КодЗапроса int, Колонка varchar(50),
                       ЗаголовокКолонки varchar(50), КлючURL varchar(50),
                       Порядок int, Формат varchar(2000), Тип int
                       + НОВОЕ: УчаствуетВБыстромПоиске tinyint

dbo.ClayGridUserParams: КодНастройкиКлиента int, Параметр varchar(20),
                        Значение varchar(1000),
                        UNIQUE (КодНастройкиКлиента, Параметр)
                        + триггер INSTEAD OF INSERT делает upsert
                          → приложение отправляет ТОЛЬКО INSERT, никогда UPDATE/MERGE
```

Тип колонки — **числовой код в `Тип int`**. В seed-данных грида №140 встречаются: `1` (КодИсследования — целое), `2` (Название — строка), `3` (ДатаСоздания — дата, `Формат='dd.MM.yyyy'`), `5` (КодТипа — справочник, в `Формат` лежит `SELECT`), `7` (Активно — логическое, `Формат='Активно=1'`). **Полный перечень кодов найти в коде** (шаг QS1), не додумывать.

## ⚠️ Ловушки — прочитать до кода

**1. `Параметр varchar(20)` — всего 20 символов.** Ключ параметра должен гарантированно влезать. Составлять его из имён колонок (`varchar(50)`) нельзя. Подробности и правило — QS5.

**2. `Значение varchar(1000)`** — список имён колонок ограничен по длине. Молча обрезать нельзя. Правило — QS5.

**3. Нельзя определять наличие колонки через `try/catch` по `SqlException`.** `DbManager` автоматически перехватывает `SqlException` и вызывает `ISqlErrorHandler.HandleSqlError()`, после чего баннер `ClayErrorBar` в `MainLayout` **покажет пользователю ошибку БД**. То есть «поймать ошибку и решить, что колонки нет» приведёт к ложному красному баннеру на экране. Наличие колонки определять **предварительной проверкой метаданных** — QS3.

**4. Поиск уже реализован — не писать второй.** У `<ClayGrid>` **уже есть** параметр `SearchColumns`, серверный поиск и покрытый тестами SQL-билдер (`tests/Clayzor.Lib.Web.Controls.Tests`, поиск выполняется на стороне SQL Server, не в памяти). Задача динамического режима — **наполнить существующий механизм** списком колонок, а не создать новый движок поиска. Любая новая реализация поиска = провал по Simplicity First.

**5. `ClayColumnSettingsDialog` — общий диалог.** Он используется и статическими гридами, и как выбор колонок для печати/Excel (`ShowSorting=false`). Новый чекбокс не должен появляться там, где он бессмысленен — QS8.

**6. Скрытое, но активное условие поиска — это баг.** Если пользователь снял последнюю галку, а строка поиска осталась непустой, данные останутся отфильтрованными невидимым условием. Правило — QS6.

**7. Триггер `INSTEAD OF INSERT`.** Сохранение — только `INSERT`. Написать `UPDATE`/`MERGE` = сломать штатный upsert.

## Definition of Done (итог по всем шагам)

- [ ] Колонки `УчаствуетВБыстромПоиске` нет → поля поиска нет, ошибок и красных баннеров нет.
- [ ] Все значения `0`/`NULL` → поле поиска скрыто.
- [ ] Есть хотя бы одна `1` у допустимого типа → поле поиска видно, поиск работает.
- [ ] Поиск идёт **на стороне SQL Server** через существующий механизм `SearchColumns`; новый движок поиска не написан.
- [ ] Недопустимые по типу колонки не участвуют в поиске, даже если админ поставил `1`.
- [ ] Выбор пользователя сохраняется в `ClayGridUserParams` как **список имён колонок**, переживает перезагрузку страницы и перезапуск приложения.
- [ ] Ключ параметра ≤ 20 символов, значение ≤ 1000 символов, сохранение — только `INSERT`.
- [ ] Статические гриды и режим выбора колонок для печати/Excel **не изменились**.
- [ ] `dotnet build` и `dotnet test` — зелёные. Новые public-члены документированы.
- [ ] Диф минимален: каждая изменённая строка прослеживается к требованию.

## Вне рамок

- Расширенный синтаксис поиска (кавычки, `AND`/`OR`, маски, несколько слов как отдельные условия).
- Полнотекстовый индекс, `CONTAINS`, `FREETEXT`.
- Третье значение `УчаствуетВБыстромПоиске` (сейчас только `0`/`1`).
- Поиск в статических гридах — он уже есть, его не трогаем.
- Админский UI для редактирования `ClayGridColumns`.
- Обновление `AGENTS.md`, `docs/clay-grid.md` — перечислить в отчёте, что стоило бы дописать, но не править.
