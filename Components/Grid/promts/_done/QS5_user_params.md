# QS5 — Сохранение выбора пользователя в `ClayGridUserParams`

> Требует `QS1`, `QS3`, `QS4`. Правила — в `QS0`.
> Здесь два жёстких ограничения схемы, которые нельзя нарушить: `Параметр varchar(20)` и `Значение varchar(1000)`.

## Требование заказчика

- Выбор пользователя сохраняется в `UserParamsTable` = `dbo.ClayGridUserParams`.
- **В параметр сохраняются только названия колонок, участвующих в поиске.**
- Константа-префикс — **`QuickSearchParamPrefix`**.

## Формат хранения

Одна строка на грид и пользователя:

| Поле | Содержимое |
| --- | --- |
| `КодНастройкиКлиента` | идентификатор пользователя — брать **тем же способом**, что и существующие пользовательские параметры (QS1, вопрос 13) |
| `Параметр` | `QuickSearchParamPrefix` + идентификатор запроса грида |
| `Значение` | имена колонок (`ClayGridColumns.Колонка`), участвующих в поиске, через запятую, без пробелов |

Пример: `Параметр = 'QS_140'`, `Значение = 'КодИсследования,Название'`.

Колонки, которые **не** участвуют, в значении не перечисляются — заказчик указал сохранять только участвующие.

### Ключ `Параметр` — ≤ 20 символов

`Параметр varchar(20)`. Правила:

1. Формат ключа — **по фактической конвенции существующих параметров** (QS1, вопрос 11–12). Если существующие ключи уже включают `КодЗапроса` — сделать так же. Если нет — выяснить, почему коллизий между гридами не возникает, и **спросить**, прежде чем выбирать формат.
2. Значение константы `QuickSearchParamPrefix` подобрать коротким (ориентир — 3–4 символа, например `QS_`), чтобы `префикс + КодЗапроса` гарантированно влезал в 20 символов.
3. **Категорически нельзя** строить ключ из имени колонки: `Колонка` — `varchar(50)`, ключ будет обрезан или запрос упадёт.
4. Добавить проверку в коде: если итоговый ключ длиннее 20 символов — это ошибка программиста, а не пользователя. Бросить `InvalidOperationException` с внятным текстом на этапе формирования ключа. Плюс unit-тест на границу.

### `Значение` — ≤ 1000 символов

- Перед сохранением проверять длину составленной строки.
- Если не влезает — **молча обрезать запрещено** (потеря настроек пользователя без предупреждения). Сообщить пользователю через существующий механизм уведомлений (`Snackbar`, как в валидации диалога настройки колонок) текстом на русском и настройку не сохранять.
- Не выдумывать сжатие, хэши, JSON — простой список имён через запятую.

## Сохранение

- **Только `INSERT`.** На таблице стоит триггер `TR_ClayGridUserParams_Upsert` (`INSTEAD OF INSERT`), который сам делает update-or-insert. `UPDATE`, `MERGE`, «сначала `DELETE`, потом `INSERT`» — **запрещены**, это ломает штатный механизм.
- Все значения — через Dapper-параметры. При сыром SQL — `commandType: CommandType.Text`.
- SQL-константу задокументировать `///` + `--` на каждую колонку.
- Момент сохранения: **применение диалога настройки колонок** (кнопка «Применить»), вместе с остальными настройками этого диалога. Не сохранять на каждый клик по чекбоксу.
- Если пользователь снял все галки — сохранять **пустое значение** (`''`), а не удалять строку. Пустая строка означает осознанный выбор «поиск не нужен» и должна перебивать админские значения `1`. Разница «нет строки» vs «пустая строка» здесь принципиальна — см. QS6.

## Загрузка

- Читать один раз при загрузке определения грида, вместе с остальными пользовательскими параметрами (не отдельным лишним рейс-трипом, если существующий код читает параметры пачкой).
- Разбор значения: split по запятой, отбросить пустые элементы, сравнение имён колонок — **регистронезависимое** (`OrdinalIgnoreCase`).
- Имена, которых больше нет в `ClayGridColumns` (админ переделал запрос), **игнорировать молча** — без ошибок и без чистки таблицы.
- Строки нет → пользователь настройку не менял → действуют админские значения (QS6).

## Чего НЕ делать

- Не переносить существующие параметры на новый формат, не переименовывать чужие ключи.
- Не создавать отдельную таблицу/колонку под эту настройку.
- Не добавлять кэш пользовательских настроек в статику/синглтон.
- Не чистить «мусорные» записи в `ClayGridUserParams`.

## → verify

1. Unit-тесты: сборка ключа (включая границу 20 символов), сериализация/десериализация списка имён, регистронезависимость, игнор неизвестных имён, пустое значение.
2Ручная проверка: изменить набор галок → «Применить» → в БД появилась/обновилась **одна** строка с ожидаемым `Параметр`/`Значение`.
3. Повторное изменение → та же строка обновилась (дубликатов нет, `UNIQUE` не нарушен).
4. F5 страницы и полный перезапуск приложения → выбор пользователя сохранился.
5. Снять все галки → «Применить» → в БД пустое `Значение`, строка на месте.
6. `dotnet build` + `dotnet test` зелёные.
