# GE — печать и выгрузка в Excel динамического грида (оркестратор)

План реализации фичи. Правила работы — те же, что в `_readme_grid_dynamic.md` (разделы «Как агент
работает с КАЖДЫМ промтом» и «Общие правила»). **Один файл = один заход = один коммит.
Не забегай вперёд. Не делай два файла за раз.**

Этот план реализует то, что `GF15_disable_dynamic_export.md` отключил, и то, что записано в
`GF7_backlog.md`, пункт 2. **GE6 откатывает GF15.**

написано, что проверять, а что нельзя.

## Главная проблема плана: генераторы работают через рефлексию

**Прочитай это до GE1, иначе не поймёшь, зачем он нужен.**

`ClayGridPrintHtmlGenerator.Build` и `ClayGridExcelGenerator.ExportToExcel` принимают
`Type entityType` и делают:

```csharp
var propMap = BuildPropertyMap(entityType);   // рефлексия: [Column("Имя")] → PropertyInfo
...
if (propMap.TryGetValue(sqlName, out var prop))
{
    var value = prop.GetValue(entity);        // entity = detailRow.Item
    SetCellValue(cell, value, prop.PropertyType);
}
```

Динамическая строка — `ClayDynamicRow`, то есть словарь `колонка → значение`. У неё нет свойств
с атрибутом `[Column]`. `BuildPropertyMap` вернёт пустую карту → **все ячейки в печати и Excel
будут пустыми.** Это не «мелкая доработка генератора»: значение ячейки достаётся принципиально
иначе.

Вторая половина проблемы: даже достав значение из словаря, генератор форматирует его по CLR-типу
(`FormatCellValue` / `SetCellValue`). Он ничего не знает про типы динамических колонок:

- Тип 5 (Список) — в данных КОД, в гриде НАИМЕНОВАНИЕ из `_dynamicLookups`;
- Тип 9 (Пиктограмма) — в данных код, в гриде картинка, в текстовом выводе нужен тултип;
- Тип 10/13 (Дата-время/время локальное) — в данных UTC, в гриде время клиента (`_clientOffset`);
- Тип 4 (Ссылка), Тип 8 (HTML), Тип 12 (Текст с ограничением) — свои правила из `Формат`.

То есть печать динамического грида, сделанная «в лоб», покажет коды вместо наименований и UTC
вместо локального времени — печатная форма разойдётся с тем, что человек видит на экране.
Это хуже, чем пустые ячейки: неверные данные выглядят как верные.

**Решение плана**: вынуть «как получить значение ячейки» из генераторов в абстракцию
`IClayGridCellReader`. Статический режим отдаёт реализацию поверх рефлексии (поведение
не меняется ни на байт), динамический — поверх словаря и своих типов.

## Что переиспользуется как есть

- `ClayGridPrintHtmlGenerator` — вся вёрстка, встроенный CSS, `@media print`, групповые строки.
  Меняется только способ получения значения ячейки (GE1).
- `ClayGridExcelGenerator` — вся стилизация ClosedXML, Excel Outline (сворачивание групп),
  авто-ширина. То же самое.
- `ClayGrid.ExportMenu.cs` — диалог выбора колонок (`ResolveExportColumnsAsync`), спиннер,
  `_isExporting`, снекбары. Меняются только шесть строк `if (DataLoader is null) return;`.
- `ClayGroupingEngine` — для группированного экспорта (GE3). Целиком, как в GG.
- JS-модули `clayGridPrint` и `clayGridExcel` — не трогаются вообще.
- `ExcelExportRequest`, `ExcelExportMode`, `BuildFilterDescription()`, `BuildGroupDescription()` —
  режим-агностичны, не трогаются.

**Фича включается только в GE6.** До него меню печати/Excel в динамическом режиме скрыто
(GF15), и написанный код лежит мёртвым. Так каждый шаг компилируется и не ломает работающий
грид. В каждом промте есть свой блок «Проверка», выполнимый на этом шаге, — не пытайся
«включить пораньше».

**GE1 — самый опасный шаг плана**: он трогает код, которым пользуется работающий статический
режим. Его проверка целиком про то, что в статике ничего не изменилось.

## Модель доверия

Экспорт подставляет в SQL: `SelectSql` (из `Запросы.SQL`), `_dynamicDef.IdColumn`
(из `Запросы.ID`), имена группировочных колонок (из `_groupColumns`). Всё это — справочник
администратора, но `IdColumn` и группировочные колонки обязаны проходить белый список
`_dynamicKnownColumns` / `_columnBySqlName` перед подстановкой в текст запроса. ID выбранных
записей идут параметрами (`@sid0`, `@sid1`, …) — как в статике, так и оставь.
