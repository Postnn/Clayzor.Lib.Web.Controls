# G3. Фикс регресса из G2: «съехали колонки» после включения full-width строки группы

Чинит регресс, введённый CSS-частью G2. C#/Razor-часть G1 и G2 (`GroupRowHostKey`, `CellClassFunc`,
класс `group-header-cell` на хост-ячейке) **не трогаем** — по HTML-дампу видно, что она уже работает
верно: у строки-заголовка ровно одна ячейка `class="mud-table-cell group-header-cell"`, остальные
четыре — пустые `<td>`. Проблема только в CSS-правилах, добавленных в `app.css`.

Файл: `app.css` (в потребляющем веб-проекте).

## Симптом

После группировки по «Тип»/«Название» заголовок таблицы съезжает: подписи колонок («Код», «Порядок»,
«Группа», «Заключение») перестают совпадать с реальными данными под ними, в шапке видны обрывки текста
не на своих местах.

## Причина

CSS из G2 делал так:

```css
.mud-table-row:has(.group-header-cell) {
    display: flex; /* ← вот это */
}
.mud-table-row:has(.group-header-cell) > .mud-table-cell:not(.group-header-cell) {
    display: none !important; /* ← и это */
}
```

В HTML-таблице ширины колонок вычисляются согласованно **по всем строкам сразу** (`<thead>` и
`<tbody>` вместе) — это то, что держит заголовки выровненными с данными. Как только `display` строки
меняется с `table-row` на `flex`, эта строка перестаёт участвовать в общем вычислении ширин колонок
наравне с остальными — браузер пересчитывает ширины по оставшимся строкам иначе, и шапка (`<thead>`,
которая осталась обычной table-строкой) рассинхронизируется с телом таблицы. Чем больше строк-групп на
странице, тем заметнее эффект (как раз то, что видно на скриншоте — списки с частой группировкой).

Это системный риск любого варианта «превратить одну строку таблицы в flex/grid» — сам приём для
раскрытия «плашки на всю ширину» рабочий, но **менять `display` у `<tr>`/`<td>` внутри `<table>` нельзя**,
если в той же таблице есть другие строки с обычной table-раскладкой (шапка, строки детализации).

## Решение

Не искать заменителей colspan через layout-хаки. Вместо этого — оставить всем `<td>` их обычный
`display: table-cell`, а тексту заголовка группы разрешить **визуально вылезать** за пределы своей узкой
ячейки поверх соседних (гарантированно пустых благодаря G1) ячеек той же строки. Раскладка таблицы не
меняется ни на пиксель — меняется только то, что рисуется поверх нее.

### Правка `app.css`

Заменить блок, добавленный в G2 (`display:flex` / `display:none` / `flex:1 1 auto` и т.д.), на:

```css
/* ── Group Header Rows: полная ширина в интерактивном гриде ──
   ВАЖНО: не менять display у <tr>/<td> — это ломает общую раскладку колонок
   таблицы (thead рассинхронизируется с tbody, см. G3). Вместо этого подпись
   группы визуально "вылезает" за пределы своей ячейки поверх соседних пустых
   ячеек той же строки (G1 гарантирует, что они пустые). Табличная раскладка
   не меняется вообще. */

.mud-table-row:has(.group-header-cell) {
    background-color: rgba(5, 22, 77, 0.04) !important;
    border-top: 2px solid var(--lh-navy) !important;
    cursor: pointer;
}

.mud-table-row:has(.group-header-cell):hover {
    background-color: rgba(5, 22, 77, 0.08) !important;
}

/* Единый фон и без вертикальных границ у всех ячеек строки —
   чтобы пустые соседние td не выглядели отдельными прямоугольниками */
.mud-table-row:has(.group-header-cell) > .mud-table-cell {
    background-color: transparent !important;
    border-right: none !important;
}

/* Хост-ячейка: разрешаем подписи вылезать вправо поверх пустых соседей.
   Ширина/структура таблицы не затрагивается — только то, что видно поверх неё. */
.mud-table-row:has(.group-header-cell) .group-header-cell {
    overflow: visible !important;
    white-space: nowrap;
    position: relative;
    z-index: 1;
}
```

Класс `.clay-group-pin-cell`, если он был добавлен в G2 для колонки чекбокса, — убрать: он был нужен
только для варианта с `display:none`, при overflow-bleed прятать нечего, колонка чекбокса остаётся как
обычная ячейка слева от хост-ячейки и не участвует в схлопывании вовсе.

### Что проверить отдельно

- `.mud-table-cell` из библиотеки MudBlazor может задавать собственный непрозрachный фон (не только из
  `app.css`) — если после патча в пустых ячейках всё ещё виден белый/другой прямоугольник поверх фона
  строки, добавить более специфичный override для этого свойства именно у
  `.mud-table-row:has(.group-header-cell) > .mud-table-cell`.
- Проверить, что `overflow-x` контейнера грида (`.mud-table-container` / скролл-обёртка) не обрезает
  вылезающий текст самых длинных названий групп при развёрнутом гриде на узком экране — если обрезает,
  это не баг раскладки таблицы, а обычное поведение горизонтального скролла, можно жить с этим или
  сократить `white-space: nowrap` до `text-overflow: ellipsis` для конкретно этого случая.

## Критерии

- [ ] Заголовки колонок (`Код`, `Порядок`, `Группа`, `Заключение`) снова совпадают по горизонтали со
      своими данными в строках детализации — при любом количестве групп на странице.
- [ ] Строка-заголовок группы визуально выглядит одной полосой на всю ширину (подпись читается целиком,
      не обрезана шириной колонки редактирования/первой колонки данных).
- [ ] Между чекбоксом (если включён режим выбора) и подписью группы нет визуальных разрывов/рамок.
- [ ] Многоуровневая группировка (2+ уровня, см. приложенный HTML — COVID-19 → РНК коронавируса →
      строка детализации) — отступы по уровням не ломают ни раскладку, ни выравнивание колонок.
- [ ] Никаких `display: flex` / `display: none` на `<tr>`/`<td>` в новых правилах не осталось.
- [ ] Обычные строки данных выглядят как раньше.
- [ ] Ручная проверка в браузере: изменить ширину окна, прокрутить грид по горизонтали — колонки не
      «плавают» независимо от того, развёрнуты группы или нет.
