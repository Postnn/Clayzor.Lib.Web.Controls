# G4. Фикс G3: колонка растягивается под самое длинное название группы

G3 не сработал: `overflow: visible` + `white-space: nowrap` на ячейке внутри таблицы с
`table-layout: auto` не создаёт «наплыв» текста поверх соседних ячеек — вместо этого браузер учитывает
полную ширину нерастягиваемого (`nowrap`) контента при расчёте **собственной** ширины этой колонки
(в auto-раскладке таблиц статичное переполнение — часть intrinsic-ширины ячейки, а не свободный пиксель
поверх соседей). Результат ровно тот, что на скриншоте: колонка с хост-ячейкой стала шириной с самое
длинное название группы на странице, а не осталась узкой с наплывающим текстом.

C#/Razor-часть (G1, класс `group-header-cell` на хост-ячейке) не трогаем — проблема только в CSS.

Файл: `app.css`.

## Решение

Чтобы контент не влиял на ширину своей колонки вообще, его нужно вынуть из нормального потока —
`position: absolute` внутри `position: relative`-ячейки. Абсолютно спозиционированный элемент не
участвует в расчёте intrinsic-ширины родителя, поэтому колонка останется узкой (как определено другими
строками — например, шириной иконки редактирования), а подпись будет свободно рисоваться поверх
соседних пустых ячеек той же строки.

### Правка `app.css`

Заменить содержимое правила `.mud-table-row:has(.group-header-cell) .group-header-cell` из G3 (и
связанные с ним) на:

```css
/* ── Group Header Rows: полная ширина в интерактивном гриде ──
   position:absolute вместо статичного overflow — абсолютно спозиционированный
   элемент не учитывается при расчёте ширины своей колонки (в отличие от
   overflow:visible + white-space:nowrap, см. G3), поэтому колонка остаётся
   узкой, а подпись свободно рисуется поверх соседних пустых ячеек строки. */

.mud-table-row:has(.group-header-cell) {
    background-color: rgba(5, 22, 77, 0.04) !important;
    border-top: 2px solid var(--lh-navy) !important;
    cursor: pointer;
}

.mud-table-row:has(.group-header-cell):hover {
    background-color: rgba(5, 22, 77, 0.08) !important;
}

/* Единый фон и без вертикальных границ у всех ячеек строки —
   чтобы пустые соседние td не выглядели отдельными прямоугольниками,
   когда подпись из хост-ячейки накроет их поверх */
.mud-table-row:has(.group-header-cell) > .mud-table-cell {
    background-color: transparent !important;
    border-right: none !important;
}

/* Хост-ячейка — точка отсчёта для абсолютного позиционирования подписи.
   Ширина самой ячейки (и всей колонки) остаётся такой, как задана колонке —
   контент внутри неё больше на неё не влияет. */
.mud-table-row:has(.group-header-cell) .group-header-cell {
    position: relative;
}

/* Сама подпись (шеврон + текст + счётчик) — вынута из потока, поэтому
   не участвует в расчёте ширины колонки и может свободно наплывать
   на соседние (гарантированно пустые благодаря G1) ячейки строки. */
.mud-table-row:has(.group-header-cell) .group-header-cell > div {
    position: absolute;
    top: 0;
    bottom: 0;
    left: 0;
    display: flex;
    align-items: center;
    white-space: nowrap;
    z-index: 1;
}
```

`> div` — это тот самый `<div role="group" class="d-flex flex-row align-center gap-3" style="padding-left:...">`,
в который уже обёрнуты шеврон и подпись (см. дамп из предыдущего сообщения) — переносить разметку не
нужно, только CSS.

### Почему это должно сработать, а прошлый вариант — нет

- **G3** держал текст в нормальном потоке (`overflow: visible` на статично расположенном блоке) — браузер
  всё равно закладывает полную ширину `nowrap`-контента в intrinsic-ширину ячейки при `table-layout: auto`.
  Отсюда рост колонки.
- **G4** убирает подпись из потока (`position: absolute`) — у элемента вне потока нет intrinsic-ширины,
  которая могла бы повлиять на родителя. Ячейка и колонка остаются такими же узкими, как заданы
  (например, шириной иконки редактирования), а подпись просто рисуется поверх — ровно то же самое,
  что происходит и с абсолютно спозиционированными бейджами/тултипами внутри обычных ячеек таблиц.

## Что проверить

- Высота строки не должна «схлопнуться»: `<td>` теперь визуально пустой изнутри (весь контент абсолютный),
  но реальная высота строки определяется остальными (непустыми) ячейками той же строки — если высота
  всё же поехала, добавить самой `.group-header-cell` `min-height`, равный высоте обычной строки грида
  (посмотреть текущую высоту dense-строки в `app.css`/MudBlazor).
- `z-index: 1` должен быть выше, чем у соседних `<td>` (обычно `auto`, т.е. ниже) — если пустые соседние
  ячейки почему-то перекрывают подпись, поднять `z-index` ещё выше.
- Клик по шеврону (переключение раскрытия группы) должен по-прежнему работать через всю визуальную
  ширину подписи, а не только в границах исходной узкой ячейки — `position:absolute` внутри `<td>` не
  меняет обработчики кликов на самой кнопке, но стоит перепроверить визуально после патча.

## Критерии

- [ ] Ширина колонки-хоста (обычно колонка редактирования) не зависит от длины названий групп — остаётся
      узкой на любой странице данных.
- [ ] Подпись группы (шеврон + название + счётчик) полностью читается на всю ширину грида, не обрезана.
- [ ] Заголовки колонок (`Код`, `Название`, `Порядок`, `Группа`, `Заключение`) по-прежнему совпадают по
      горизонтали со своими данными в строках детализации.
- [ ] Многоуровневая группировка — вложенные подписи со своим `padding-left` не съезжают и не обрезаются.
- [ ] Клик по шеврону разворачивает/сворачивает группу как раньше.
- [ ] Обычные строки данных выглядят как раньше.
- [ ] Ручная проверка в браузере: длинные названия групп (например, «КТ органов брюшной полости с
      внутривенным контрастным усилением»), узкое окно, горизонтальный скролл.
