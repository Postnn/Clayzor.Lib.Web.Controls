# GB17 — в диалоге настраиваемого фильтра появились два скролла

Прочитать перед началом: `Components/Grid/Filter/ClayFilterDialog.razor` — целиком, особенно
`<MudDialog ContentClass="clay-filter-dialog-content">`, `<DialogContent>`, вложенный
`<div style="flex:1 1 auto;overflow-y:auto;min-height:0">` и инлайновый `<style>` с
`.clay-filter-dialog-content { overflow:hidden; display:flex; flex-direction:column }`;
`Components/Grid/Filter/ClayFilterGroup.razor` и `.razor.cs` — есть ли там свой скролл-контейнер;
`app.css` (обе копии) — `.clay-filter-*`; для сравнения — как этот же паттерн (`ContentClass`
+ `pa-0`) сделан в GB4 (`ClayColumnSettingsDialog`), если GB4 к этому моменту выполнен.

## Дефект (гипотеза — подтвердить в DevTools)

Диалог задуман с ОДНИМ скроллом: `.clay-filter-dialog-content` (лежит на `.mud-dialog-content`
через `ContentClass`) имеет `overflow:hidden` и `display:flex; flex-direction:column`, а
прокручивается внутренний `<div style="…overflow-y:auto…">` с деревом условий. Пока
`.mud-dialog-content` не скроллится сам, скролл один.

Наиболее вероятная причина двух полос: `.mud-dialog-content` MudBlazor по умолчанию имеет
`overflow-y:auto` и собственный `padding`. Если `overflow:hidden` из инлайнового `<style>`
перестал применяться (перебит специфичностью правил MudBlazor, порядком подключения CSS после
GB7, или `ContentClass` не доехал до нужного узла), то скроллятся сразу оба контейнера:
и `.mud-dialog-content`, и внутренний `div`. Симптом «после GB15» — совпадение по времени
(GB15 правил только `ClayGrid.Filtering.cs`, разметку и CSS диалога он не трогал), либо GB15
был выполнен агентом с побочной правкой разметки. **Проверь фактический диагноз до правок.**

## Диагностика (обязательный первый шаг)

Открой диалог на стенде, в DevTools найди, у КАКИХ элементов реально появилась вертикальная
полоса (проверяй `scrollHeight > clientHeight` и вычисленный `overflow-y`). Возможные исходы:

1. Скроллятся `.mud-dialog-content` и внутренний `div` одновременно → `overflow:hidden` на
   `.clay-filter-dialog-content` не сработал. Правка — п. A.
2. Скроллятся внутренний `div` и `.clay-filter-group--root`/иной вложенный контейнер → лишний
   `overflow` появился ниже по дереву (возможно, внесён при выполнении GB15/иного шага). Правка —
   п. B.
3. Что-то иное → зафиксируй в отчёте фактическую картину и правь по тому же принципу: скролл
   ровно один, на внутреннем `div` дерева условий.

Не переходи к правке, пока не назвал в отчёте конкретные два элемента со скроллом.

## Изменить/создать

### A. Если скроллится `.mud-dialog-content` (исход 1)

Перенести стили контента из хрупкого инлайнового `<style>` в `app.css`/`clay.css` классом и
гарантировать `overflow:hidden` + отсутствие паддинга у `.mud-dialog-content` именно этого
диалога. По образцу GB4 (`ContentClass="clay-column-settings-content pa-0"`):

`ClayFilterDialog.razor`:

```razor
<MudDialog ContentClass="clay-filter-dialog-content pa-0">
```

и убрать инлайновый `<style>` из низа файла, перенеся его правила в CSS (в `clay.css` после GB7,
иначе в ОБЕ копии `app.css`):

```css
/* Контент диалога фильтра: единственный скролл — на внутреннем контейнере дерева
   условий, поэтому сам .mud-dialog-content не скроллится и без паддинга. */
.clay-filter-dialog-content {
    width: 600px;
    max-width: 100%;
    height: min(460px, 80vh);
    overflow: hidden;
    display: flex;
    flex-direction: column;
    gap: 4px;
    padding: 0 !important;   /* перебивает дефолтный паддинг .mud-dialog-content */
}
```

Внутренний `<div style="flex:1 1 auto;overflow-y:auto;min-height:0">` — оставить, это законный
единственный скролл. Структурные инлайны (`flex`, `overflow-y`, `min-height`) StyleGuard
пропускает; но если удобнее — тоже вынеси в класс `.clay-filter-dialog__scroll`.

### B. Если лишний скролл ниже по дереву (исход 2)

Найти элемент с внесённым `overflow-y:auto`/`scroll` внутри дерева условий и убрать его —
прокрутка должна быть только на внешнем `div` дерева. `.clay-filter-group--root/--indented`
скроллиться не должны (у них только `display:flex`); если у них появился `overflow` — удалить.

### Общее

Если GB7 выполнен — правка CSS в `clay.css`. Если нет — в ОБЕИХ копиях `app.css` дословно
одинаково. Инлайновый `<style>` в `.razor` в любом случае убрать (он дублирует то, что должно
жить в CSS, и именно такие блоки — источник хрупкости с `overflow`).

`STYLE_RULES.md` §4: строка «Контент диалога фильтра → `.clay-filter-dialog-content`». Другую
документацию не трогать.

## Не делай

- Не откатывай GB15 — сортировка списка колонок к скроллу отношения не имеет; убедись в этом
  диагностикой, а не предположением.
- Не добавляй `overflow:hidden` глобально на `.mud-dialog-content` — сломаешь прокрутку других
  диалогов; класс вешается только на этот через `ContentClass`.
- Не убирай внутренний скроллящийся `div` — тогда длинное дерево условий вылезет за пределы
  диалога. Цель — один скролл, а не ноль.
- Не фиксируй `height` жёстко в пикселях без `min(…, 80vh)` — на низких экранах диалог не влезет.
- Не трогай `ClayColumnSettingsDialog` (его скролл — задача GB4) и другие диалоги.
- Не меняй ширину/высоту диалога «на глаз» ради маскировки скролла — правка про `overflow`,
  а не про размеры.
