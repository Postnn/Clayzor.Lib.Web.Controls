# D1 — конвенция «Options» для компонентов библиотеки

Серия CGO решила задачу для одного компонента. Этот шаг превращает решение в правило, чтобы
`ClayTreeView` (серия CT) и следующие компоненты не расходились с гридом, и чтобы через год
никто не завёл двадцать первый параметр тега.

**Кода здесь почти нет** — один класс-заготовка, правка документации и сверка двух
существующих options-классов.

## Прочитать

- `Components/Grid/ClayGridOptions.cs` (итог серии CGO);
- `Components/Tree/Models/ClayTreeOptions.cs` (серия CT, если она выполнена);
- `src/Clayzor.Lib.Web.Controls/AGENTS.md` — раздел «Настройки компонентов (Options)»,
  добавленный в C2;
- `docs/clay-grid.md` — раздел про `ClayGridOptions`.

## 1. Сверка двух классов

Пройди по `ClayGridOptions` и `ClayTreeOptions` и приведи к одному виду **то, что расходится
без причины**:

- POCO с `{ get; set; }`, `sealed`, дефолты в объявлении свойств;
- статический `Defaults`, если он нужен потребителю (у грида нужен — фолбэк в
  `ClayGridPageBase`; у дерева, возможно, нет — тогда не добавлять ради симметрии);
- xml-doc на каждом свойстве; в заголовке класса — абзац «создаётся один раз в поле страницы»;
- тематические блоки-разделители внутри плоского класса, а не вложенные объекты;
- имена одинаковых по смыслу настроек совпадают между компонентами. Проверь поимённо кандидатов
  и приведи к одному варианту: `Title`, `Id`, `PageSize`, `ShowPagination`, `PersistState`.
  Различие имён у одинаковых по смыслу настроек — то, из-за чего потом читают документацию
  вместо кода.

**Расхождения, у которых есть причина, оставляй и объясняй в отчёте.** Симметрия ради симметрии
здесь вреднее асимметрии.

## 2. Правило в `AGENTS.md`

Раздел «Настройки компонентов (Options)» (создан в C2) дополнить до полного правила — коротко,
пунктами, без рассуждений:

1. Конфигурация компонента живёт в отдельном классе `Clay<Имя>Options`, рядом с компонентом.
2. Параметрами тега остаются только: данные, меняющиеся между рендерами; `RenderFragment`;
   `EventCallback`; живые ссылки на потребителя (`DataLoader="this"`); `@ref`-совместимые
   служебные параметры.
3. Объект настроек создаётся страницей один раз (поле + `OnInitialized`), не выражением
   в разметке. **Причина, одной фразой:** новый объект на каждый рендер ломает сравнение
   ссылки на параметр.
4. Компонент внутри читает единственное поле действующих настроек, а не параметры поштучно.
5. Дефолты свойств — единственный источник значений по умолчанию; тест-защёлка на дефолты
   обязательна (образец — `ClayGridOptionsTests`).
6. Ничего, что компонент **записывает** сам, в options не кладётся.
7. Настройки уровня приложения (из `appsettings`/`web.config`, через DI) — это **другой** класс
   и другое имя (`Clay*DynamicOptions`, `ClayAppSettings`), в options экземпляра их не смешивать.

## 3. Заготовка для новых компонентов (опционально, по решению заказчика)

Если сочтёте полезным — файл-образец `docs/component-options-template.md` с минимальным
скелетом: класс options, параметр `Options`, поле `_opt`, разрешение в двух точках жизненного
цикла, пример разметки страницы. Это не абстрактный базовый класс и не интерфейс
`IClayOptions` — **их заводить не надо**: у options-классов нет общего поведения, общий у них
только приём использования, а приёмы живут в документации, а не в наследовании.

## 4. Вопрос заказчику: пороги пересмотра

Сформулируй в отчёте (не решай сам) два вопроса, которые серия CGO оставила открытыми:

- **Группировка по секциям.** `ClayGridOptions` — 21 свойство. При ~30 плоский класс станет
  неудобным, и появится смысл в `Options.Data.SelectSql` / `Options.Toolbar.ShowPrint`.
  Переход механический (IDE-рефакторинг), но это ломающее изменение. Делать заранее или ждать
  порога?
- **Переименования.** В `ClayGridOptions` перенесены неудачные имена: `Id` (это DOM-id, а не
  идентификатор грида — путается с `DynamicGridId`), `Dynamic` (флаг режима, читается как
  прилагательное), `SelectVisible` (речь о выборе строк, а не о видимости SELECT).
  Переименовать сейчас, одним ломающим изменением, или оставить?

## Не делай

- Не заводи базовый класс/интерфейс для options-классов и не тащи их в отдельную сборку.
- Не меняй `ClayGridOptions` и `ClayTreeOptions` по существу: этот шаг — про единообразие
  оформления и правило в документации, а не про новые настройки или переименования
  (переименования — отдельное решение, см. п. 4).
- Не переписывай серию CT и не начинай новых компонентов.
- Не правь документацию за пределами `AGENTS.md` библиотеки и (если создаётся) нового файла
  в `docs/`.

## Проверка

- `dotnet build Clayzor.sln` + `dotnet test` — зелёные;
- оба options-класса открыты рядом и различия объяснимы: каждое несовпадение либо устранено,
  либо перечислено в отчёте с причиной;
- правило в `AGENTS.md` читается за минуту и состоит из проверяемых утверждений
  («создаётся в поле страницы»), а не из советов («старайтесь выносить настройки»);
- если создан `docs/component-options-template.md` — его скелет компилируется: собери по нему
  пустой компонент в черновике, убедись, что шаблон полон, черновик удали;
- ручной чек-лист CGO0 прогонять не нужно, если код компонентов не менялся; если менялся —
  прогнать целиком.
