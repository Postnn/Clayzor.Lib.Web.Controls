# C1 — удалить устаревшие параметры тега

Ломающее изменение публичного API компонента. Все потребители в решении уже переведены
(B1–B4), внешних (NuGet-пакета, других решений) у библиотеки нет — поэтому переходный период
с `[Obsolete]` не нужен: он оставил бы в коде два способа настройки, и следующий разработчик
выбрал бы старый.

**Перед началом — точка откота** (git-коммит по указанию заказчика). Это шаг, после которого
«вернуть как было» одной правкой уже не получится.

## Прочитать

- отчёты **A1** (список MOVE) и **B4** (перечень потребителей и неявных зависимостей);
- `Components/Grid/ClayGrid.razor.cs` — `Options`, `_opt`, `ResolveOptions`,
  `CollectLegacyConflicts`;
- `Components/Grid/ClayGridOptions.cs` — `Defaults`, `DiffFromDefaults`, `Clone`.

## Что сделать

**1. Удалить 21 объявление `[Parameter]`** (точный список — из A1, группа MOVE).
Удаляются вместе с докстрингами — тексты уже живут в `ClayGridOptions`.

**2. Упростить `ResolveOptions`:**

```csharp
private ClayGridOptions ResolveOptions() => Options ?? new ClayGridOptions();
```

Ветка сборки из legacy-параметров и метод `CollectLegacyConflicts` удаляются целиком —
это орфаны, созданные шагом A3, их чистка входит в объём (`/AGENTS.md`: убираем за собой).

**3. Решить судьбу `DiffFromDefaults`** (появился в A4 ради тестируемости конфликтов).
После удаления конфликтов у него нет вызывающих. Варианта два, выбери и обоснуй в отчёте:
- удалить вместе с его тестами — если больше нигде не нужен (**предпочтительно**, Simplicity First);
- оставить как публичную утилиту — **только** если в B2/B3/B4 нашёлся реальный второй
  потребитель. «Пригодится» не считается.

`Defaults` — **оставить**: он используется в `ClayGridPageBase` как фолбэк
(`Grid?.Options ?? ClayGridOptions.Defaults`, шаг B1).

`Clone()` — оставить, если его кто-то вызывает (проверь grep'ом); если нет — удалить вместе с
докстрингом. Не оставляй «на будущее»: неиспользуемый публичный метод в библиотеке — это
обещание, которое никто не поддерживает.

**4. `Options` перестаёт быть nullable-опциональным?** Нет — оставь `ClayGridOptions?` и
фолбэк на новый объект. Грид без настроек (все дефолты) — законная конфигурация, а
`[EditorRequired]` заставил бы задавать `Options` даже в тестовой разметке. Одна строка
в докстринге: не задан — используются значения по умолчанию.

## Не делай

- Не помечай удаляемые параметры `[Obsolete]` «на один релиз» — переходного периода нет
  по решению заказчика (см. преамбулу шага).
- Не удаляй STAY-параметры (`Items`, `Loading`, `TotalCount`, `PageNumber`, `Columns`,
  `ColumnDefs`, `DataLoader`, `OnAdd`, `OnGroupToggle`, `OnQueryChanged`) — они остаются
  навсегда, это не техдолг.
- Не переименовывай свойства `ClayGridOptions` в этом шаге (D1) — соблазн «раз уж всё равно
  ломаем API» велик, но тогда диф C1 перестанет быть проверяемым: сейчас он состоит только
  из удалений.
- Не меняй логику, не оптимизируй, не трогай `_opt`-обращения из A3.
- Не удаляй `Defaults`.

## Проверка

**Механическая**
- `dotnet build Clayzor.sln` — зелёный. **Если упал — это удача**: значит, B4 пропустил
  потребителя. Исправляй потребителя (перевод на `Options`), а не возвращай параметр;
- `dotnet test` — зелёный (тесты `DiffFromDefaults` либо удалены вместе с методом, либо
  проходят);
- `grep -rn "\[Parameter\]" src/Clayzor.Lib.Web.Controls/Components/Grid/ClayGrid*.cs` →
  ровно 10 попаданий: `Options` + 9 STAY. Приложи вывод к отчёту;
- `grep -rn "CollectLegacyConflicts\|устаревш" src/Clayzor.Lib.Web.Controls/Components/Grid/`
  → ни одного попадания;
- `grep -rn "Options=\"@(new ClayGridOptions" src/` → пусто.

**Ручная — полная приёмка всей серии, оба стенда, чек-лист CGO0 целиком.**
Дополнительно, для проверки нового поведения «без настроек»:
- временно добавить в любое приложение страницу с `<ClayGrid TEntity="IClayGridRow" />`
  без `Options` → компонент рендерится с дефолтами (заголовок «Список», пустой грид),
  не падает с `NullReferenceException`; **страницу удалить после проверки**;
- в браузерной консоли и в логе приложения — ни одной новой ошибки;
- прокрутка, drag&drop колонок и сохранение состояния работают (проверка, что `Id` из опций
  доехал до DOM и JS-хелперов — риск, отмеченный в B4).

**Отчёт:** сколько параметров удалено, что стало с `DiffFromDefaults`/`Clone`, какие
потребители нашлись на этапе падения сборки (если нашлись), что осталось в бэклоге.
