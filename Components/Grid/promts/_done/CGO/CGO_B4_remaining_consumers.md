# B4 — добить остатки: кто ещё зависит от параметров грида

Шаг-ревизия перед удалением публичного API (C1). Задача: убедиться, что legacy-параметры
`ClayGrid` больше **никто** не задаёт и не читает, кроме самого механизма совместимости из A3.
Если пропустить и сразу делать C1, сборка упадёт в неожиданном месте — или, хуже, не упадёт,
потому что параметр читался через рефлексию/строку.

## Прочитать

- отчёт **A1**, список «внешние потребители»;
- отчёты **B1–B3** (что уже переведено).

## Что сделать

**1. Найди всех, кто задаёт параметры грида в разметке:**

```
grep -rn "<ClayGrid" src/ --include=*.razor
```

Для каждого найденного тега выпиши список атрибутов и проверь: все конфигурационные — в
`Options`? Кандидаты, о которых легко забыть: второй грид на той же странице, гриды внутри
диалогов, демо/тестовые страницы, закомментированная разметка (её тоже привести в порядок или
удалить — но только если она мертва, иначе в отчёт).

**2. Найди всех, кто читает настройки грида из кода:**

```
grep -rn "\.Options\b" src/ --include=*.cs --include=*.razor
grep -rn "IClayGrid" src/
grep -rn "ClayGrid<" src/ tests/
```

Обрати внимание на:
- `ClayColumn` / `ClayColumnDef` — должны читать через `Grid.Options` (B1);
- диалоги (`ClayColumnFilterDialog`, `ClayColumnValueFilterDialog`, `ClayColumnSettingsDialog`,
  `ClayFilterDialog`) — получают ли что-то из настроек грида параметрами диалога;
- генераторы (`ClayGridPrintHtmlGenerator`, `ClayGridExcelGenerator`) — им передают `Title`
  и колонки **аргументами**; проверь, что аргумент теперь берётся из `_opt.Title`, и что нигде
  не осталось второго источника заголовка;
- `ClayGridExportFileName`, `ClayGridLinkResolver` — по карте A1;
- тестовый проект: `grep -rn "ClayGrid" tests/` — если тесты конструируют настройки, перевести
  на `ClayGridOptions`.

**3. Проверь неявные зависимости — то, что grep по имени параметра не находит:**

- **рефлексия и строковые имена параметров.** `DialogParameters { ["Model"] = … }`,
  `ParameterView`, `typeof(...).GetProperty("Title")` — поищи:
  ```
  grep -rn "GetProperty(\|GetProperties(\|ParameterView\|DialogParameters" src/Clayzor.Lib.Web.Controls/
  ```
  Параметр, задаваемый по строковому имени, при удалении в C1 не уронит сборку — сломается
  в рантайме. Каждое попадание разбери и выводы — в отчёт;
- **JS-интероп по `Id`.** `grep -rn "clay-grid" src/ --include=*.js --include=*.razor --include=*.cs`
  — селекторы в JS-хелперах, ожидающие конкретный DOM-`id`. `Id` теперь приходит из опций;
  убедись, что значение то же;
- **CSS-селекторы по `id`** в `clay.css` (`#clay-grid…`), если такие есть.

**4. Приведи найденное в порядок** — механически, без переделок. Если потребитель сложный
(например, требует настройку, которой в `ClayGridOptions` нет) — **стоп, вопрос заказчику**.

## Не делай

- Не удаляй legacy-параметры (C1) и не помечай `[Obsolete]`.
- Не переводи на `Options` то, что настройками не является (`Items`, `Loading`, `TotalCount`,
  `PageNumber`, `DataLoader`, `On*`, `Columns`, `ColumnDefs`).
- Не «заодно» рефактори найденные шероховатости в диалогах и генераторах — находки в отчёт.
- Не удаляй мёртвый код, обнаруженный по пути, если он не создан твоими же правками
  (`/AGENTS.md`: чистим только своих орфанов).

## Проверка

- `dotnet build Clayzor.sln` + `dotnet test` — зелёные;
- `grep -rn "<ClayGrid" src/ --include=*.razor` → у **каждого** найденного тега атрибуты
  только из списка STAY (+ `Options`, `@ref`, `TEntity`);
- отчёт содержит полный перечень мест, где параметры грида задаются или читаются, с отметкой
  «переведено / оставлено осознанно (STAY) / требует решения заказчика»;
- перечень «неявных» зависимостей (рефлексия, строковые имена, JS по `id`, CSS по `id`) —
  с выводом по каждой: рискует ли она сломаться в C1;
- полный ручной чек-лист из CGO0 (оба стенда).
