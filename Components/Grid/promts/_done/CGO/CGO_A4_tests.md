# A4 — тесты `ClayGridOptions` и композиции настроек

Тестовый проект решения (`tests/Clayzor.Lib.Web.Controls.Tests`) намеренно проверяет только
**чистую детерминированную логику**: без БД и без рендеринга Blazor-компонентов (bUnit в
решении нет и заводить его в этом шаге не надо). Значит, тестируем то, что можно вызвать без
компонента, а вызываемость и определяет, как устроен код.

## Прочитать

- `tests/Clayzor.Lib.Web.Controls.Tests/README.md` — «Что покрывается» и «Как добавить тест»;
- любой существующий файл тестов как образец стиля, например `ColumnTypeRegistryTests.cs`;
- `Components/Grid/ClayGridOptions.cs` (A2) и `ResolveOptions`/`CollectLegacyConflicts` (A3).

## Задача

Проверяемого без компонента здесь два куска: **дефолты** и **обнаружение конфликтов**.

### 1. Дефолты — `ClayGridOptionsTests.cs`

Тест, фиксирующий значения по умолчанию: `Title == "Список"`, `Id == "clay-grid"`,
`PageSize == 50`, `ShowAddButton`, `ShowPagination`, `AllowColumnReorder`,
`EnableValueFilter` — `true`, `ColumnMenuMode == ColumnMenuMode.Mobile`,
`EditSuccessMessage == "Запись обновлена"`, `SelectVisible`/`ShowPrint`/`ShowExcel`/`Dynamic`
— `false`, `SearchColumns` — пустой непустой-ссылкой массив, `FilterColumnTypes` — пустой
непустой-ссылкой словарь, `EditDialogType`/`FilterLookupOptions`/`CustomBatchGroups`/
`DynamicGridId` — `null`.

Смысл теста не в проверке компилятора, а в **защёлке**: этот список — контракт совместимости
со всеми страницами, которые не задают параметр явно. Случайная правка дефолта здесь падает
в CI, а не у тестировщика. Так и напиши в комментарии к классу — иначе следующий читатель
удалит «бессмысленный» тест.

`ClayGridOptions.Defaults` — тем же тестом: `Defaults` возвращает тот же набор значений и
**тот же экземпляр** при повторном обращении.

### 2. Обнаружение конфликтов

`CollectLegacyConflicts()` сейчас — приватный метод компонента, вызвать его из теста нельзя.
Не делай его `public` и не выставляй `internal` + `[assembly: InternalsVisibleTo]` ради теста:
метод переезжает в C1 (после удаления legacy-параметров он исчезнет вовсе). Вместо этого
вынеси **чистую** часть логики в статический хелпер, который переживёт C1 условно и
тестируется без компонента:

```csharp
/// <summary>
/// Сравнивает настройки с эталоном значений по умолчанию и возвращает имена свойств,
/// заданных отлично от дефолта. Чистая функция.
/// </summary>
public static IReadOnlyList<string> DiffFromDefaults(ClayGridOptions options)
```

Место — `ClayGridOptions.cs` (статический метод рядом с `Defaults`). Компонент в
`CollectLegacyConflicts` собирает `ClayGridOptions` из своих legacy-параметров и передаёт
результат в `DiffFromDefaults` — то есть в A3-коде остаётся только сборка объекта, а сравнение
становится тестируемым. **Правку `CollectLegacyConflicts` под этот хелпер сделай в этом же
шаге** (это единственное отступление от «A3 закончен»; больше в грид не лезь).

Тесты `DiffFromDefaults`:
- объект-дефолт → пустой список;
- изменён один `Title` → ровно `["Title"]`;
- изменены три разных свойства → три имени, порядок не проверяется (`Assert` по множеству);
- `SearchColumns = []` (новый пустой массив, другая ссылка, то же содержимое) → **не** конфликт;
- `SearchColumns = ["A"]` → конфликт;
- `FilterColumnTypes` — новый пустой словарь → не конфликт; словарь с элементом → конфликт;
- `PageSize = 50` (равно дефолту, но задано явно) → **не** конфликт. Это осознанное
  ограничение приёма «сравнение с дефолтом»: явно заданное дефолтное значение неотличимо от
  незаданного. Тест фиксирует его как ожидаемое поведение, а не как баг; в докстринге
  `DiffFromDefaults` — та же фраза.

## Не делай

- Не подключай bUnit, не рендери `ClayGrid` в тестах, не пиши тесты на `ResolveOptions`
  напрямую — компонент в этом проекте не тестируется, и менять это решение в рамках
  рефакторинга нельзя.
- Не делай приватные члены компонента `public`/`internal` ради тестируемости.
- Не пиши тесты «на всякий случай» вокруг `Clone()`, если он в коде ещё не используется, —
  проверяй его в том шаге, где он реально понадобится (B2/B3).
- Не трогай существующие тесты. Если какой-то из них перестал компилироваться — это регрессия
  A3, останавливайся и разбирайся с ней, а не правь тест.

## Проверка

- `dotnet test tests\Clayzor.Lib.Web.Controls.Tests` — зелёный, новые тесты видны в выводе;
- в новом файле нет обращений к БД, `DbManager`, `IJSRuntime`, `RenderTreeBuilder`;
- намеренно испортить один дефолт в `ClayGridOptions` (например `PageSize = 25`) →
  тест дефолтов падает; **вернуть**;
- намеренно сломать `DiffFromDefaults` (вернуть всегда пустой список) → падают тесты
  конфликтов; **вернуть**;
- `dotnet build Clayzor.sln` — зелёный; ручной чек-лист из CGO0 прогонять не нужно
  (кода грида, кроме `CollectLegacyConflicts`, не касались) — но убедись, что проверка
  конфликта из блока «Проверка» шага A3 всё ещё воспроизводится.
