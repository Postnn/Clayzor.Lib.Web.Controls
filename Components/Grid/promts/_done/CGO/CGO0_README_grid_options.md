## Зачем

У `ClayGrid<TEntity>` сейчас ~30 параметров тега. Каждая новая функция добавляла ещё один,
и разметка страницы превратилась в двадцатистрочный список атрибутов, где вперемешку лежат
конфигурация (`SelectSql`, `ShowPrint`), связывание данных (`Items`, `TotalCount`) и
обработчики. Следствия, которые уже мешают:

- **Конфигурация не переиспользуется.** Два грида с одинаковыми настройками — два копипаста
  разметки. Нельзя вынести «профиль грида» в поле, метод или в общий код.
- **Каждая фича — публичное API.** Добавление флага меняет сигнатуру компонента; серия GB/GF/GE
  добавила `SelectVisible`, `ShowPrint`, `ShowExcel`, `EnableValueFilter`, `Dynamic`,
  `DynamicGridId` — все шесть теперь навсегда в публичном контракте.
- **`IClayGrid` протекает.** Интерфейс отдаёт `SelectSql`/`SearchColumns`/`DefaultOrder`/
  `EditDialogType` поштучно, и `ClayGridPageBase` в ~10 местах пишет
  `Grid?.SelectSql ?? string.Empty`.
- **Новые компоненты уже строятся иначе.** `ClayTreeView` (серия CT) настраивается через
  `ClayTreeOptions`. Нужен один приём на библиотеку, а не два.

Цель серии: **конфигурация — в одном объекте `ClayGridOptions`; параметрами тега остаётся
только то, что параметром обязано быть.** С ~30 атрибутов до 9.

---

## Что переезжает, а что нет — читай до кода

Это главное архитектурное решение серии. Blazor-параметры делятся на три сорта, и смешивать
их в одном объекте нельзя.

### Остаются `[Parameter]` (9)

| Параметр | Почему нельзя в options |
|---|---|
| `Items` | меняется на каждой загрузке данных |
| `Loading` | меняется дважды на каждую загрузку |
| `TotalCount` | приходит со страницы после каждого запроса |
| `PageNumber` | синхронизация пагинатора при авто-переходах в `ToggleGroup` |
| `Columns` | `RenderFragment` — это дочерняя разметка, а не настройка |
| `ColumnDefs` | то же |
| `DataLoader` | живая ссылка на страницу (`DataLoader="this"`) |
| `OnAdd`, `OnGroupToggle`, `OnQueryChanged` | `EventCallback`: Blazor сам обеспечивает ре-рендер вызывающего компонента после обработчика. В POCO это теряется |

**Причина запрета для первых четырёх — не вкусовая.** Если положить `Items`/`TotalCount` в
options, страница будет обязана пересоздавать объект настроек на каждый рендер
(`Options="@(new ClayGridOptions { Items = _rows, … })"`), грид на каждом `OnParametersSet`
увидит новую ссылку, и любая логика «параметр изменился» начнёт срабатывать вхолостую.
Отсюда правило, действующее во всех шагах серии: **объект options создаётся один раз в поле
страницы, а не выражением в разметке.**

### Переезжают в `ClayGridOptions` (21)

`Title`, `Id`, `SelectSql`, `SearchColumns`, `DefaultOrder`, `PageSize`, `ShowAddButton`,
`ShowPagination`, `AllowColumnReorder`, `ColumnMenuMode`, `SelectVisible`, `ShowPrint`,
`ShowExcel`, `EnableValueFilter`, `EditDialogType`, `EditSuccessMessage`, `FilterColumnTypes`,
`FilterLookupOptions`, `CustomBatchGroups`, `Dynamic`, `DynamicGridId`.

Точный список с типами и значениями по умолчанию агент **проверяет по коду** на шаге **A1** —
таблица выше собрана по `ClayGrid.razor.cs` и `docs/clay-grid.md` и может отставать от
реальности.

## Зафиксированные решения (не пересматривать)

1. **Рефакторинг behavior-preserving.** Ни одного изменения поведения на всех шагах A и B.
   Никакой новой валидации, которая раньше не бросала исключений (кроме конфликта
   `Options` + legacy-параметр, п. 4). Никаких переименований.
2. **Стадия A ничего не переименовывает.** Свойства `ClayGridOptions` называются **ровно** как
   текущие параметры (`Dynamic`, `DynamicGridId`, `Id`), даже где имя так себе. Переименования
   и группировка по секциям — только после C1, отдельным решением заказчика (см. D1).
3. **Плоский набор свойств, а не вложенные секции** (`Data`/`Toolbar`/`Filtering`). На 21
   свойстве вложенность не окупается, а стадию A из механической замены превращает в
   проектирование. Так же устроен `ClayTreeOptions` — библиотека остаётся однородной.
   Порог для пересмотра — ~30 свойств (D1).
4. **Стадия A обратно совместима.** Legacy-параметры остаются публичными и рабочими; внутри
   грид работает только с собранным объектом. Если задан и `Options`, и legacy-параметр,
   отличающийся от значения по умолчанию → `InvalidOperationException` со списком конфликтующих
   имён. Молчаливого приоритета быть не должно.
5. **Внутри грида остаётся ровно один источник истины — поле `_opt`.** Ни один partial-файл
   после A3 не читает миграционные параметры напрямую. Проверяется grep'ом.
6. **`IClayGrid` отдаёт `Options` целиком**, а не поштучные `SelectSql`/`SearchColumns`/
   `DefaultOrder`/`EditDialogType` (шаг B1).
7. **Осторожно с именем.** В решении уже есть `ClayGridDynamicOptions` — это **DI-настройки
   динамического режима** (имена таблиц, префиксы параметров, байнд из `appsettings`).
   Новый `ClayGridOptions` — **настройки одного экземпляра грида на странице**. Разные сущности
   с похожими именами: в xml-doc каждого класса обязана быть ссылка на другой и одна фраза
   про разницу.
8. **Опции — read-mostly.** Грид читает `_opt.X` в точке использования, поэтому правка
   свойства извне подействует при следующем рендере. Исключение — `PageSize`: он и сейчас
   снимается один раз в `OnInitialized` (`_pageSize = PageSize`); это поведение сохраняется.
   Никаких событий «опции изменились» не вводим.

## Общий регрессионный чек-лист

Прогоняется **после каждого шага**, начиная с A3. Рефакторинг behavior-preserving: любое
расхождение — дефект шага, а не «улучшение».

**Сборка**
- `dotnet build Clayzor.sln` — зелёный, без новых warning'ов;
- `dotnet test tests\Clayzor.Lib.Web.Controls.Tests` — зелёный.

**Статический режим — `Clayzor.App.Web.MedicalTests`, `/medical-tests`**
- грид открылся, данные на месте, заголовок и кнопка «Добавить» как были;
- поиск, сортировка по клику (ASC → DESC → снять), пагинация и смена размера страницы;
- перетаскивание заголовка в трей группировки → группы, счётчики, раскрытие/сворачивание,
  авто-переход страницы при сворачивании;
- перетаскивание заголовка в трей фильтрации → диалог колоночного фильтра → чип; фильтр по
  значению (Excel-style); настраиваемый составной фильтр;
- «Настройка колонок»: видимость, порядок, drag&drop;
- «Выбрать записи» → чекбоксы, tri-state в шапке и на группах, счётчик «Выбранные (N)»;
- «Печать» и «Выгрузка в Excel» — все три режима каждая (текущая страница / выбранные / все);
- сервисная колонка с карандашом → диалог редактирования → снекбар → перезагрузка;
- кнопка справки на тулбаре (`URI_help_clayGrid`) открывается в новом окне.

**Динамический режим — `Kesco.App.Web.Inventory`, `/?id=140`**
- грид грузится сразу, без ручного «Обновить»;
- сохранение и восстановление состояния (колонки, сортировка, группировка, размер страницы,
  фильтр) — уйти со страницы и вернуться;
- кнопка «Выбрать записи» есть (заполнен `Запросы.ID`), выбор работает;
- печать и Excel во всех режимах; индикатор `.clay-grid-busy` на долгих операциях;
- негативный: `/` без `?id=` → сообщение «не указан код запроса», а не пустой грид.

**Общее**
- тёмная тема; печать (`Ctrl+P`) — треи и тулбар скрыты;
- в консоли браузера нет новых ошибок;
- `grep -rn "Options=\"@(new ClayGridOptions" src/` → **ни одного попадания** (объект options
  создаётся в поле страницы, не в разметке).

## Не входит в серию

- Рефакторинг `ClayColumnDef`/`ClayColumn` на options — у них 9 и 2 параметра, и они
  дочерние компоненты-регистраторы; смысла нет.
- Разбор того, что грид пишет в свои же `[Parameter]` (`Items`, `TotalCount` в
  `ClayGrid.Dynamic.cs`) — известный смелл, отдельная задача. **В этой серии не исправлять**,
  но занести в отчёт.
- Вложенные секции опций, переименования, `Validate()` с семантическими проверками
  (`Dynamic` + `DataLoader` одновременно и т. п.) — см. D1, решение заказчика.
- Приведение `ClayGridDynamicOptions` к новому имени.
