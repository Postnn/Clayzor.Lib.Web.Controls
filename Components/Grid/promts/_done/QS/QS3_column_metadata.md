# QS3 — Наличие колонки и чтение флага в метаданные

> Требует выполненного `QS1`. Правила — в `QS0`. Это самый рискованный шаг: здесь ломается обратная совместимость.

## Цель

Прочитать `УчаступетВБыстромПоиске`… **нет, точное имя — `УчаствуетВБыстромПоиске`** (скопируй имя из `QS0`, не набирай по памяти) в метаданные колонки динамического грида так, чтобы **отсутствие этой колонки в `ColumnsTable` не приводило ни к исключению, ни к сообщению об ошибке на экране**.

## Что сделать

### 1. Имя колонки — один раз

Добавить `УчаствуетВБыстромПоиске` в `ColumnNames.cs` (по конвенции: имена колонок БД определяются ровно один раз). Ссылаться на константу оттуда. SQL-константы от этого правила освобождены — но продублированного строкового литерала в C#-коде быть не должно.

### 2. Определение наличия колонки

**Способ — предварительная проверка метаданных**, до основного запроса:

```sql
-- Наличие опциональной колонки быстрого поиска в таблице определений колонок.
-- COL_LENGTH возвращает NULL, если колонки нет. Совместимо с SQL Server 2008 R2.
SELECT CASE WHEN COL_LENGTH(@table, @column) IS NULL THEN 0 ELSE 1 END
```

- Имя таблицы передавать **параметром** (`@table`), значение брать из того же источника, что и основной запрос определений колонок (см. ответ на вопрос 7 из QS1). Не хардкодить `dbo.ClayGridColumns`, если в коде имя таблицы конфигурируемо.
- Результат вычислять **один раз на загрузку определения грида** и хранить в модели определения (например `bool ПоддерживаетБыстрыйПоиск` / `SupportsQuickSearch` — имя согласуй со стилем соседних свойств). Не звать проверку на каждый рендер и не на каждую колонку.

### 3. Основной запрос определений колонок

Дальше — по факту из QS1, вопрос 5:

- **Если запрос использует явный перечень колонок:** держать **две** версии SQL-константы — с колонкой и без неё — и выбирать по результату проверки из п.2. Обе задокументировать `///` + `--` на каждую колонку. Никакой склейки SQL из фрагментов «на лету» ради экономии — читаемость важнее.
- **Если запрос использует `SELECT *`:** дополнительная версия не нужна; читать значение из результата, только когда п.2 дал `1`.

### 4. Свойство в модели колонки

Добавить в модель метаданных колонки свойство под флаг. Тип — **`bool`**, не `tinyint`/`byte`/`int`: наружу это признак «участвует / не участвует».

Правила преобразования, зафиксировать в `///`-комментарии:
- `1` → `true`
- `0` → `false`
- `NULL` → `false`
- колонки в таблице нет → `false` для всех колонок
- **любое другое числовое значение** (`2`, `7`, …) → `false`, без исключения и без записи в лог как об ошибке. Сейчас допустимы только `0` и `1`; расширение значений — вне рамок.

## Ловушка (повтор из QS0, здесь она срабатывает)

**Не определять наличие колонки через `try/catch` вокруг запроса.** `DbManager` сам перехватывает `SqlException` и вызывает `ISqlErrorHandler.HandleSqlError()` → `ClayErrorBar` в `MainLayout` покажет пользователю красный баннер с текстом ошибки БД. Даже если исключение будет проглочено выше, баннер уже будет показан. Поэтому «попробовать и посмотреть, упадёт ли» — **неприемлемый** способ проверки наличия колонки.

## Чего НЕ делать

- Не кэшировать результат проверки между разными гридами/запросами глобально (статические словари, синглтоны) — храни в модели определения текущего грида.
- Не читать `INFORMATION_SCHEMA` циклом по каждой колонке.
- Не менять сигнатуры существующих публичных методов, если можно добавить свойство в модель.
- Не делать флаг конфигурируемым параметром компонента.

## → verify

1. Колонка **есть**: значения `1/0/NULL` из `ClayGridColumns` корректно превращаются в `true/false/false` в модели.
2. Колонка **удалена** (`ALTER TABLE dbo.ClayGridColumns DROP COLUMN УчаствуетВБыстромПоиске`): грид открывается, данные грузятся, `ClayErrorBar` **не появляется**, в консоли нет `SqlException`, флаг у всех колонок `false`.
3. Колонка возвращена — поведение как в п.1.
4. Unit-тест на преобразование значений (`1/0/NULL/2` → `true/false/false/false`).
5. `dotnet build` + `dotnet test` зелёные.
