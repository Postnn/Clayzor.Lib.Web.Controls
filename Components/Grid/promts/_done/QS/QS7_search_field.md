# QS7 — Поле поиска на панели управления грида

> Требует `QS6`. Правила — в `QS0`.

## Главное правило шага

**Поиск уже реализован.** У `<ClayGrid>` есть параметр `SearchColumns`, серверный поиск и покрытый тестами SQL-билдер; поиск, сортировка, группировка и пагинация по конвенции выполняются на стороне SQL Server. Задача — **подать в существующий механизм** набор колонок, вычисленный в QS6.

Если по итогам шага в дифе появился новый класс поиска, свой `WHERE`-билдер или фильтрация в памяти (`.Where(...)` по загруженным строкам) — шаг выполнен неверно, переделать.

## Что сделать

1. **Наполнить `SearchColumns`** в динамическом режиме итоговым набором из QS6. Формат значения — ровно тот, который ожидает существующий механизм (установлен в QS1, вопрос 1). Пересчитывать при загрузке определения грида и при применении настроек колонок.

2. **Поле ввода.**
   - Если поле поиска на панели управления **уже есть** (QS1, вопрос 2) — переиспользовать его, добавив только условие видимости из QS6. Новое поле не создавать.
   - Если поля нет — добавить `MudTextField` с `Variant.Outlined` (стандарт формы в проекте), в общем ряду элементов панели управления, с иконкой `Icons.Material.Filled.Search`, placeholder на русском («Поиск»), кнопкой очистки (`Clearable`), `Immediate="true"` и `DebounceInterval` порядка 300–400 мс.
   - Панель управления обёрнута в `div` с `flex-wrap` — проверить, что поле не ломает перенос на узких экранах.
   - Кнопки с тултипом — только `<ClayButton>`; выпадающие меню — только `<ClayMenu>` (конвенция проекта).

3. **Реакция на ввод** — тем же путём, которым поиск работает сейчас: обновление свойств `_query` (не переприсваивание объекта — иначе `TotalCount` сбрасывается в 0 при async-рендере) и перезагрузка данных.

4. **Загрузка данных** — по конвенции в `OnAfterRenderAsync(bool firstRender)` с `if (firstRender)`, **не** в `OnInitializedAsync`. Ничего из этого менять не нужно, просто не сломать.

5. **Видимость** — по правилам QS6. Скрытие — именно скрытие, не `Disabled`.

## Требования к SQL-условию

Детали и тесты — в QS9, здесь только то, что нельзя нарушить при подаче колонок:

- Условие поиска комбинируется с фильтрами и группировкой через `AND` и **обязательно оборачивается в скобки**: `... AND (col1 LIKE ... OR col2 LIKE ...)`. Без скобок `OR` «съест» остальные условия и поиск начнёт возвращать лишние строки — классическая ошибка, проверить на гриде с активным фильтром.
- Значение из поля поиска передаётся **только Dapper-параметром**. Подстановка в строку SQL запрещена.
- Только синтаксис SQL Server 2008 R2.

## Чего НЕ делать

- Не менять поведение поиска в статических гридах.
- Не добавлять кнопку «Найти» — поиск по вводу с debounce.
- Не добавлять историю поиска, автодополнение, подсветку найденного, счётчик совпадений.
- Не сбрасывать группировку/фильтры/сортировку при вводе в поиск.
- Не реализовывать поиск по нескольким словам как отдельные условия (вне рамок, QS0).

## → verify

1. Набор непуст → поле видно; ввод текста фильтрует данные; очистка возвращает полный список.
2. Поиск работает **вместе** с активным фильтром и с включённой группировкой; результаты корректны (проверить на данных, где без скобок был бы виден лишний набор строк).
3. Пагинация: результат поиска корректно постранично листается, `TotalCount` соответствует найденному, а не полному набору.
4. Поиск + режим выбора: выделение сбрасывается при изменении строки поиска (существующее поведение, не дублировать код).
5. Узкий экран ≈600px и мобильный вид — панель управления не разъезжается.
6. Тёмная тема — поле не выбивается по стилю.
7. `dotnet build` + `dotnet test` зелёные.
