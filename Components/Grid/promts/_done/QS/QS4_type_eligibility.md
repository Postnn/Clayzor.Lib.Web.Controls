# QS4 — Допустимость колонки по типу

> Требует `QS1`, `QS3`. Правила — в `QS0`.

## Требование заказчика

> Участвовать в быстром поиске могут только колонки тех типов, которые имеют **явное представление в источнике данных**.

## Что это значит (единственно верное прочтение)

Быстрый поиск — это `WHERE` по исходному запросу грида на стороне SQL Server. Значит искать можно только там, где **то, что пользователь видит в ячейке, физически лежит в результате исходного SQL**.

Не подходят колонки, у которых видимый текст появляется **после** выборки:

- **Справочник** (в seed это `Тип=5`, `КодТипа`): в источнике лежит числовой код, а пользователь видит наименование из другого запроса, указанного в `Формат`. Искать по тому, что видно, в исходном запросе невозможно.
- **Логическое/вычисляемое по выражению** (в seed это `Тип=7`, `Активно` с `Формат='Активно=1'`): в ячейке галка/значок, текста для подстроки нет.

Подходят колонки, значение которых берётся из источника напрямую: строки, целые/дробные числа, даты.

## Что сделать

1. По результатам QS1 (вопросы 9–10) составить **белый список кодов `Тип`**, допустимых для быстрого поиска. В seed-данных ориентир: `1`, `2`, `3` — допустимы; `5`, `7` — нет. **Остальные коды классифицировать по фактическому поведению из кода, а не по номеру.**
2. Список определить **в одном месте**, рядом с перечислением типов колонок. Один статический метод вида:

```csharp
/// <summary>
/// Определяет, может ли колонка динамического грида участвовать в быстром поиске.
/// Допустимы только типы, значение которых берётся напрямую из исходного запроса грида
/// (строки, числа, даты). Справочные и вычисляемые типы исключены: отображаемый текст
/// формируется после выборки и не может быть найден условием WHERE по источнику.
/// </summary>
internal static bool МожетУчаствоватьВБыстромПоиске(int типКолонки)
```

Имя метода/параметров — **в стиле соседнего кода** (если рядом всё по-английски, назови по-английски; смешивать не надо).

3. **Правило приоритета, зафиксировать в коде и в комментарии:** недопустимый тип **перебивает** и админское значение `1`, и выбор пользователя. Колонка недопустимого типа не участвует в поиске никогда.

4. Ни при каких условиях не пытаться «дотянуться» до справочного значения: никаких `JOIN` со справочником из `Формат`, никаких подзапросов по нему, никакого поиска по коду вместо наименования. Это отдельная большая задача и она **вне рамок**.

## Чего НЕ делать

- Не добавлять в `ColumnsTable` новых колонок для описания допустимости — тип уже есть.
- Не делать белый список конфигурируемым (параметр компонента, appsettings, БД).
- Не выводить пользователю сообщений о том, что колонка недопустима, — в UI это решается видом чекбокса (QS8).

## → verify

1. Unit-тесты: по одному кейсу на каждый код `Тип` из фактического перечня — допустим/недопустим.
2. Unit-тест приоритета: колонка типа справочника с `УчаствуетВБыстромПоиске=1` (в стенде это `КодТипа`, см. QS2) **не** попадает в набор колонок поиска.
3. В отчёте — итоговая таблица «код типа → расшифровка → допустим да/нет → почему». Если хоть по одному коду уверенности нет — **спросить**, а не угадывать.
