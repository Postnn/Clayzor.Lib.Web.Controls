# CGH — Кнопка вызова пользовательской документации на тулбаре ClayGrid

> Промт для агента. Стиль исполнения — по корневому `AGENTS.md`:
> **Think Before Coding → Simplicity First → Surgical Changes → Goal-Driven Execution.**
> Весь пользовательский текст — **на русском**. Каждый новый public/protected член — с `/// <summary>`.
> Git (add/commit/push) — **только по прямому указанию**. `AGENTS.md` и docs **не обновлять** без отдельной просьбы.

---

## Задача (требования заказчика)

1. На панели инструментов `ClayGrid` добавить кнопку, открывающую **пользовательскую документацию по гриду**.
2. Кнопка — **самая правая** на панели инструментов.
3. Кнопка — **в общем стиле остальных кнопок тулбара** (иконка + тултип, те же размеры/классы, никакого выбивающегося вида).
4. Документация открывается **в отдельном окне** браузера (новая вкладка/окно), текущая страница грида остаётся как есть.
5. Адрес документации берётся из параметра с именем **`URI_help_clayGrid`**.
6. Значение параметра задаётся **в `web.config` приложения, в стандартной секции `appSettings`**.
7. И приложение, и грид получают это значение **через сборку `Clayzor.Lib.Web.Settings`**.

## Контекст

- Стек: .NET 10 Blazor Server (Interactive Server), MudBlazor 9.x. Грид — `Clayzor.Lib.Web.Controls` (Razor Class Library).
- Цепочка зависимостей уже позволяет гриду читать настройки:
  `Clayzor.Lib.Web.Controls ← Entities + Web.Settings + MudBlazor`. **Новую зависимость добавлять не нужно.**
- `BindClaySettings()` в `Clayzor.Lib.Web.Settings/ClayAppSettings.cs` уже подмешивает `web.config` через `AddWebConfig()` рядом с `appsettings.json`.
- Приложение-потребитель — веб-проект решения (`Clayzor.App.Web.MedicalTests` в текущем `Clayzor.sln`; если целевое приложение называется иначе — **подтверди фактическое имя по `Clayzor.sln`** и работай с ним).

## ⚠️ Три ловушки — прочитай до кода

**1. Стиль кнопок тулбара.** В `AGENTS.md` есть два правила, которые формально противоречат друг другу:
- общее: «кнопки с тултипом — всегда `<ClayButton>`, а не `<MudTooltip><MudIconButton/></MudTooltip>`»;
- про тулбар грида: «кнопки тулбара используют `MudIconButton` с CSS-классами `grouping-toggle-btn` / `filter-toggle-btn` / `toolbar-add-btn` / `toolbar-select-btn` / `toolbar-batch-btn` и тултипами».

**Не решай это по памяти. Открой `ClayGrid.razor` и посмотри, как реально написаны соседние кнопки тулбара, и повтори именно этот приём.** Требование «в общем стиле других кнопок» означает буквально: как соседи, а не как тебе кажется правильным.

**2. «Отдельное окно» ≠ `NavigationManager.NavigateTo`.** `NavigateTo` откроет документацию **вместо** текущей страницы. Нужно новое окно: либо анкор (`Href` + `Target="_blank"` + `rel="noopener"`), либо `window.open` через JS-interop. **Приоритет — вариант без нового JS-файла** (Simplicity First). Проверь, поддерживает ли используемый в тулбаре компонент `Href`/`Target`; если нет — см. Шаг 4.

**3. Не сломай другие потребители грида и тесты.** `ClayGrid` живёт в RCL, её используют разные хосты (включая `tests/Clayzor.Lib.Web.Controls.Tests`), где настройки могут быть не зарегистрированы. Если внедрить настройки как **обязательную** зависимость — упадёт DI там, где её нет. Внедряй так, чтобы отсутствие настройки давало **безопасное поведение**, а не исключение.

## Шаг 1 — Параметр в `web.config`

В `web.config` приложения, в стандартную секцию `appSettings`, добавить ключ **точно** с этим именем:

```xml
<appSettings>
  <add key="URI_help_clayGrid" value="https://пример/docs/clay-grid" />
</appSettings>
```

- Имя ключа — **`URI_help_clayGrid` буква в букву**, не переименовывать «под C#-конвенции», не транслитерировать (по аналогии с правилом «имена колонок БД не переводим»).
- **Не** дублировать параметр в `appsettings.json` — заказчик указал `web.config`.
- Значение `value` — заглушка-плейсхолдер; реальный адрес подставит администратор.

**→ verify:** приложение запускается, `web.config` остаётся валидным XML.

## Шаг 2 — Чтение параметра через `Clayzor.Lib.Web.Settings`

В `Clayzor.Lib.Web.Settings` добавить свойство настроек для этого адреса — **минимально, тем же механизмом, каким там уже читается `ConnectionString`** (не выдумывай второй, параллельный способ конфигурации).

- **Решено заказчиком, обсуждению не подлежит:** ключ конфигурации — `URI_help_clayGrid`, C#-свойство — **`UriHelpClayGrid`**. Свойство привязывается к ключу `URI_help_clayGrid` и снабжается `/// <summary>`, в котором прямо указано: значение читается из секции `appSettings` файла `web.config`, ключ `URI_help_clayGrid`. Ни ключ, ни свойство не переименовывать.
- Значение по умолчанию при отсутствии ключа — пустая строка/`null` (не бросать исключение, не подставлять зашитый URL).
- **Никакой** валидации/нормализации URL, ретраев, кеширования — этого не просили.

**→ verify:** unit-тест в `tests/Clayzor.Lib.Web.Controls.Tests` (или подходящем тестовом проекте): ключ задан → свойство содержит значение; ключ отсутствует → пусто и **без исключения**. `dotnet test` зелёный.

## Шаг 3 — Кнопка на тулбаре `ClayGrid`

- Грид получает адрес **через DI из `Clayzor.Lib.Web.Settings`** (сборка уже в зависимостях). Внедрение — **необязательное** (nullable / `[Inject(Required = false)]`-эквивалент или проверка на `null`), чтобы хосты без зарегистрированных настроек не падали (ловушка №3).
- **Не** добавлять `[Parameter]` для URL и не заставлять каждую страницу его передавать — заказчик сказал: грид берёт значение через `Web.Settings`.
- Разметка кнопки — **копия приёма соседних кнопок тулбара** (тот же компонент, та же схема тултипа), плюс свой CSS-класс в общем ряду: `toolbar-help-btn` (оформить в `wwwroot/css/app.css` рядом с `toolbar-add-btn` / `toolbar-select-btn`, используя MudBlazor-переменные `--mud-palette-*` для авто-адаптации к тёмной/светлой теме — как сделано у соседей).
- Иконка: `Icons.Material.Filled.HelpOutline`. Тултип (русский): **«Документация»** (допустимо «Справка по таблице» — выбери одно и не плоди варианты).
- **Позиция — самая правая в панели инструментов**: последняя в разметке тулбара. Учти, что тулбар обёрнут в `div` с `flex-wrap` — проверь визуально, что кнопка остаётся справа и **не ломает перенос элементов на узких экранах**. Если для прижатия вправо нужен `margin-left: auto`, применяй его так, чтобы `flex-wrap` не разъезжался.

**→ verify:** кнопка видна последней справа на тулбаре, визуально неотличима по стилю от соседних, тултип на русском, на узком экране (≈600px) верстка не ломается.

## Шаг 4 — Открытие в отдельном окне

Предпочтительный вариант — **без JS**: кнопка рендерится как ссылка с `Href="<значение настройки>"`, `Target="_blank"`, `rel="noopener"`.

- Если компонент кнопок тулбара **не** поддерживает `Href`/`Target` — выбери **один** из двух путей и **обоснуй выбор в отчёте**:
  a) минимально расширить `ClayButton` параметрами `Href`/`Target` (аккуратно, не меняя текущее поведение существующих вызовов);
  b) `window.open(url, '_blank', 'noopener')` через JS-interop.
  При варианте (b) — **не создавай новый .js-файл ради одной строки**, положи helper в один из существующих RCL-скриптов (`_content/Clayzor.Lib.Web.Controls/js/...`) и не забудь, что новые скрипты требуют регистрации в `App.razor` (лишней регистрации избегаем).
- **Запрещено**: `NavigationManager.NavigateTo(url)` без нового окна, `forceLoad`, замена текущей страницы, открытие документации в модальном диалоге/`iframe` внутри приложения.

**→ verify:** клик открывает адрес из `web.config` в **новой вкладке/окне**; исходная страница с гридом остаётся живой, состояние грида (сортировка/фильтры/страница) не сбрасывается, circuit не переподключается.

## Шаг 5 — Поведение, если параметр не задан

Если `URI_help_clayGrid` отсутствует или пуст — **кнопка не отображается** (никаких битых ссылок, `about:blank`, ошибок в консоли, Snackbar-ов).

**→ verify:** убери ключ из `web.config` → кнопки нет, остальной тулбар работает как раньше; верни ключ → кнопка появилась.

## Критерии приёмки (Definition of Done)

- [ ] Кнопка есть, она **последняя справа** на тулбаре `ClayGrid`, стилистически идентична соседним кнопкам, тултип на русском.
- [ ] Документация открывается **в отдельном окне**; текущая страница и состояние грида не затронуты.
- [ ] Адрес читается из **`web.config` → `appSettings` → `URI_help_clayGrid`** (имя ключа буква в букву) **через `Clayzor.Lib.Web.Settings`**; в коде URL не зашит; в `appsettings.json` не дублируется.
- [ ] C#-свойство настройки называется **`UriHelpClayGrid`** и документировано ссылкой на ключ `URI_help_clayGrid`.
- [ ] Грид получает значение через DI из `Web.Settings`; новых зависимостей у проектов не появилось; `[Parameter]` для URL не добавлялся.
- [ ] Ключ отсутствует/пуст → кнопка скрыта, исключений нет; хосты без зарегистрированных настроек (тесты) не падают.
- [ ] Остальные кнопки, порядок тулбара, верстка и поведение грида **не изменились** (regression не допущен).
- [ ] Новые public/protected члены документированы `/// <summary>`; CSS — в общем ряду с `--mud-palette-*`.
- [ ] Диф минимален: каждая изменённая строка прослеживается к требованию; соседний код не «улучшался».
- [ ] Git-коммит не сделан; `AGENTS.md` и `docs/` не изменены.

## Вне рамок (не делать)

- Написание самой пользовательской документации, страниц справки внутри приложения, `iframe`-просмотрщика.
- Контекстная справка (разные URL для разных страниц/гридов), якоря, локализация ссылки, подстановка параметров в URL.
- Добавление пункта «Документация» в мобильное меню колонки (`⋮`) — только если попросят отдельно.
- Конфигурируемость иконки/тултипа/позиции кнопки через параметры компонента.
- Обновление `AGENTS.md`, `docs/clay-grid.md`, `STYLE_RULES.md` — можешь **перечислить в отчёте**, что стоило бы дописать, но не правь.
