# F10. Скролл у выпадающего списка оператора

Файлы: `Components/Grid/Filter/ClayFilterExpression.razor`,
(опц.) `Components/Grid/Filter/ClayFilterOperatorLabels.cs`.
Отдельная косметика после F9 (тело диалога уже с одним скроллом).

## Причина (по DOM попапа)
Попап списка оператора — один контейнер `mud-popover … overflow-y-auto`
(`max-height:300px`), внутри `.mud-list` без своего overflow. Двойного скролла на
элементе нет. Полоса появляется из-за того, что попап берёт ширину поля
(`mud-popover-relative-width`, ~137px), и длинные подписи («больше или равно (≥)»,
«не пусто (NOT NULL)») переносятся в две строки → высота списка превышает 300px.
Рядом с полосой тела диалога это выглядит как «вторая полоса».

## Исправить
Убрать перенос подписей, чтобы список влезал без прокрутки.

1. Расширить поле «Условие» в `ClayFilterExpression.razor` (попап наследует ширину поля):
   - `min-width:110px` → `min-width:180px`;
   - `flex:1 1 120px` → `flex:1 1 160px`.
   Тогда попап ~180px, 8 пунктов по ~36px укладываются под 300px без скролла.

2. (Альтернатива/дополнительно) сократить подписи сравнений в
   `ClayFilterOperatorLabels`: «>», «≥», «<», «≤» вместо «больше (>)», «больше или равно (≥)»
   и т.п. — заодно уплотнит и строку условия. Полные подписи можно оставить для
   текстовых операторов (содержит/начинается с…), где неоднозначности нет.

3. (Если пунктов действительно много) поднять высоту попапа: у `MudSelect` оператора
   задать `MaxHeight="400"`.

## Критерий
- [ ] При обычном числе операторов список раскрывается без вертикальной прокрутки.
- [ ] Рядом с телом диалога нет визуально «двойной» полосы.
- [ ] `dotnet build` без ошибок.

> Примечание: прокрутка в самом списке — нормальна, если пунктов реально много.
> Цель — чтобы она не появлялась из-за переноса подписей.
