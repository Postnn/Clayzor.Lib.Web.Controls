# GN — группировка на произвольное число уровней (оркестратор)

Снятие потолка уровней группировки в `ClayGroupingEngine`. **Касается ОБОИХ режимов** —
статического (`ClayGridPageBase`) и динамического (`ClayGrid.Dynamic.Grouping`). Правила работы —
`_readme_grid_dynamic.md`, разделы «Как агент работает с КАЖДЫМ промтом» и «Общие правила».
**Один файл = один заход = один коммит.**

## Что на самом деле сломано

В `GG0_README_dynamic_grouping.md` и `GF7_backlog.md` (п. 1а) записано мягко: «третий уровень
молча игнорируется». **Разбор показал, что всё хуже.** Читай внимательно — от этого зависит,
что ты будешь делать.

### Факт 1: потолок в трёх местах, и они не согласованы

`BuildGroupAggregateSql`:

```csharp
var selectParts = new List<string>();
for (int i = 0; i < 3; i++)                       // ← ровно 3 слота
{
    selectParts.Add(i < groupExprs.Count
        ? $"{groupExprs[i]} AS K{i}"
        : $"CAST(NULL AS SQL_VARIANT) AS K{i}");  // ← добивка NULL-ами
}

var grp = string.Join(", ", groupExprs);          // ← а GROUP BY по ВСЕМ колонкам
```

`GridGroupRow` — три поля `K0`, `K1`, `K2` (Dapper мапит по именам колонок).

`BuildAggregates`:

```csharp
if (gr.K0 is not null) keys.Add(gr.K0.ToString()!);
if (gr.K1 is not null) keys.Add(gr.K1.ToString()!);   // ← K2 не читается ВООБЩЕ
```

То есть: SQL умеет 3, `GridGroupRow` умеет 3, `BuildAggregates` умеет 2, а `GROUP BY` — сколько
угодно. Четыре разных мнения о числе уровней в одном конвейере.

### Факт 2: три уровня не «не работают» — они дают неверные данные

Группировка по `[a, b, c]`. SQL: `SELECT a AS K0, b AS K1, c AS K2, COUNT(*) ... GROUP BY a,b,c`.
Пришло:

| K0 | K1 | K2 | Cnt |
|----|----|----|-----|
| a1 | b1 | c1 | 2 |
| a1 | b1 | c2 | 3 |
| a1 | b2 | c3 | 1 |

`BuildAggregates` читает только K0/K1 → первые две строки дают **один и тот же** `FullKey`
`a1\u001Fb1`. `seenKeys` защищает только синтетических родителей, листья добавляются как есть →
в `aggregates` два узла с одинаковым ключом. `BuildTree` кладёт оба в `Children` родителя `a1`.

Что видит пользователь: **две одинаковые группы «b1»** — одна «(2 шт.)», другая «(3 шт.)».
Раскрывает первую: `RawKeys = [a1, b1]` → `WHERE a=a1 AND b=b1` вернёт 5 строк, а `DetailEnd`
посчитан от `ItemCount = 2` → покажет первые 2. Раскрывает вторую: тот же `WHERE`, `ItemCount = 3`
→ покажет строки 1–3. **Строки 1 и 2 показаны дважды, строки 4 и 5 не показаны нигде.**

Дублирующиеся заголовки и пересекающаяся детализация — это не «фича не доделана», это порча
данных на экране. То же уедет в печать и Excel (GE).

### Факт 3: четыре уровня и больше — то же самое, плюс `K3` вообще не в SELECT

`GROUP BY a,b,c,d` при `SELECT` только `K0..K2` → строк агрегата больше, дублей больше.
Перетащить в лоток четвёртый чип пользователю никто не мешает: `AddGroupColumn` ограничений
не имеет.

### Факт 4: NULL в группировочной колонке ломает всё сегодня, без всяких трёх уровней

Число уровней выводится из «а ключ не null?»:

```csharp
if (gr.K0 is not null) keys.Add(...);
if (gr.K1 is not null) keys.Add(...);
var depth = keys.Count - 1;
```

Одна колонка группировки, в данных `NULL` → `K0 is null` → `keys` пуст → `depth = -1` →
`keys.Last()` на пустом списке → **`InvalidOperationException`**. Грид падает.

Две колонки, во второй `NULL` → `keys = [a1]` → `depth = 0` → узел становится КОРНЕВЫМ и
конфликтует с настоящим родителем `a1`. Снова дубли.

**Корень всех четырёх фактов один: число уровней должно приходить из числа группировочных
колонок, а не угадываться по null-ам.** Отсюда план.

## Что делаем

Уровней столько, сколько чипов в лотке. Никакого потолка: `SELECT` отдаёт `K0..K{n-1}`,
`GridGroupRow` хранит список ключей, `BuildAggregates` читает ровно `Keys.Count` уровней,
`NULL` — это законное значение ключа, а не признак отсутствия уровня.

Искусственный лимит не вводим и не обсуждаем: пользователь тянет в лоток столько колонок,
сколько нужно, и SQL Server ограничивает это сам (размер `GROUP BY`). Наш код мнения иметь
не должен.

## Порядок выполнения

| Файл | Что делает | Ломает поведение? |
|------|-----------|-------------------|
| `GN1_group_row_keys.md` | `GridGroupRow.Keys` вместо `K0/K1/K2`; общий маппер; `BuildGroupAggregateSql` без потолка | нет (1–2 уровня работают как раньше) |
| `GN2_build_aggregates_n_levels.md` | `BuildAggregates` на N уровней; NULL — законный ключ | **да, в лучшую сторону**: 3+ уровня перестают дублироваться |
| `GN3_group_key_where.md` | `BuildGroupKeyWhere`: `IS NULL` вместо `= @dk` для NULL-ключей; перевод всех мест сборки детального WHERE | да: группы с NULL перестают быть пустыми |
| `GN4_export_interleaving.md` | N уровней в C#-interleaving экспорта; согласование `FullKey`; вычистить упоминания «до 3-х уровней» | да |

**GN1 строго перед GN2**: `BuildAggregates` читает `Keys`, которого до GN1 нет.
**GN2 строго перед GN3**: `RawKeys` на N уровней появляются в GN2.
GN4 — последний, он опирается на формат `FullKey`, зафиксированный в GN2.

## Зависимости от других планов

Обязательных нет. Но **если выполнены `GG1` или `GE3`** — в них есть код, который GN трогает:

- `GG1` создал `ClayDynamicGroupMapper` (словарь → `GridGroupRow` по именам K0/K1/K2/Cnt).
  **GN1 его заменяет** общим маппером на N ключей.
- `GG2` (`LoadDynamicGroupedData`) и `GE3` (`BuildDynamicGroupedExportRows`) собирают детальный
  `WHERE` через `{exprs[i]} = @dk{i}`. **GN3 переводит их на `BuildGroupKeyWhere`.**

Если этих файлов в репозитории нет — просто пропускай их в списках правок и **скажи об этом
в отчёте**, не создавай их «заодно».

## Что НЕ трогаем

`BuildTree`, `ComputeParentCounts`, `ComputeEffectiveRows`, `WalkTree` — **уже рекурсивны и
готовы к N уровням**. Прочитай их и убедись сам: они работают с `Depth`, `ParentKey` и
`Children`, а числа 3 в них нет. Ни одной правки там не требуется. Если тебе кажется, что
требуется, — ОСТАНОВИСЬ и объясни, что именно ты там увидел.

`ClayGroupHeader.razor` (отступ `Depth * 16px`) — тоже без потолка, работает на любой глубине.
Лоток и чипы (`ClayGrid.Grouping.cs`, `ClayGrid.DragDrop.cs`) ограничений не имеют — не добавляй.

## Совместимость

`GridGroupRow` — публичный тип библиотеки, GN1 меняет его форму (`K0/K1/K2` → `Keys`).
Это осознанный слом: тип существует только как DTO строки агрегатного запроса, за пределами
`ClayGroupingEngine` и его вызывающих он не используется. **Перед GN1 проверь это сам:**
`grep -rn "GridGroupRow\|\.K0\|\.K1\|\.K2" --include=*.cs --include=*.razor` по всему решению.
Если найдёшь использование вне списка из GN1 — ОСТАНОВИСЬ и покажи.
