# G5 — кнопки из определения: редактирование, добавление, удаление

Прочитать перед началом: как в проекте открываются формы/навигация; `ConfirmDialog`
(docs/confirm-dialog.md); где резолвятся значения конфигурации (для '@').

Создать хелпер `ClayGridLinkResolver.Resolve(string? value)`:
- null/пусто → null (нет действия);
- начинается с '@' → взять значение из конфигурации по имени без '@' (через IConfiguration/
  существующий резолвер); если нет — вернуть null и залогировать;
- иначе — вернуть как есть (относительная или абсолютная ссылка).

Изменить ClayGrid (динамический режим):
1. Колонка-«карандаш» в строке видна ТОЛЬКО если `Resolve(def.EditForm)!=null`. Клик →
   переход на `edit + идентификатор строки` (значение `def.IdColumn` из строки данных).
2. Кнопка «+» видна ТОЛЬКО если `Resolve(def.NewForm)!=null`. Клик → переход на `new`.
3. Инлайн-удаление доступно, если `def.SqlDelete` не пусто: по клику показать `ConfirmDialog`;
   при подтверждении выполнить `def.SqlDelete` ТОЛЬКО через `DynamicSql.ExecuteAsync(Db,
   def.SqlDelete, new { id = <значение def.IdColumn строки> })` (Lib.Entities, G1b); затем
   обновить грид.

Не делай: не выполняй SqlDelete без подтверждения; не подставляй id конкатенацией — только `@id`;
НЕ вызывай SQL/Dapper напрямую из Controls и не создавай `DbManager` — доступ к БД строго через
`Clayzor.Lib.Entities`.

Проверка (ручная, на сиде #140):
- при заполненных EditForm/NewForm карандаш и «+» видны и ведут на '/medical/edit'/'/medical/new';
- значение вида '@SomeConfigKey' резолвится в URL из конфигурации;
- при пустом SqlDelete кнопки удаления нет; при заполненном — удаление после подтверждения
  выполняет DELETE и грид обновляется.
