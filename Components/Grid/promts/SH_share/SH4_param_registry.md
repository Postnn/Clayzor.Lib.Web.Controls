# SH4 — Единый реестр имён параметров грида

> Требует `SH1`. Правила — в `SH0`. Небольшой, но ключевой этап: без него `SH5` и `SH8` будут дублировать одну и ту же логику по-разному.

## Зачем

Три места задачи должны отвечать на один и тот же вопрос — «какие имена параметров относятся к текущему гриду»:

- `SH5` — что копировать в общую настройку;
- `SH7` — по каким именам искать общие настройки текущего грида;
- `SH8` — с чем сравнивать имена параметров, пришедших по `sharedId`.

Заказчик описывает это как «`...ParamPrefix` + идентификатор грида». Если каждое из трёх мест соберёт список по-своему, расхождение проявится не сразу: ссылка сформируется, а на открытии выдаст «несоответствие параметров». Поэтому список формируется **в одном месте**.

## Что сделать

1. Собрать все существующие константы-префиксы имён параметров (найдены в `SH1`, вопрос 2), включая `QuickSearchParamPrefix`, в **одно** место — если они уже лежат вместе, ничего не переносить, только использовать.
   Быстрый поиск на момент постановки задачи **уже реализован** (колонка `УчаствуетВБыстромПоиске` присутствует в схеме стенда вместе с seed-значениями), поэтому его параметр существует в коде реально и обязан попасть в реестр. Если в рабочей копии константа названа иначе — использовать фактическое имя и указать его в отчёте.

2. Добавить один метод, отдающий полный набор имён параметров для конкретного грида:

```csharp
/// <summary>
/// Возвращает полный набор имён параметров (ClayGridUserParams.Параметр),
/// относящихся к указанному гриду: каждый известный префикс + идентификатор запроса.
/// Единственный источник истины для копирования настроек в общую ссылку,
/// поиска общих настроек грида и проверки соответствия параметров при открытии по sharedId.
/// </summary>
internal static IReadOnlyCollection<string> GetGridParamNames(int кодЗапроса)
```

Имена метода и параметров — **в стиле соседнего кода** (если рядом всё по-английски — по-английски; смешивать языки в одном файле не надо).

3. **Проверка длины.** Каждое собранное имя обязано влезать в `Параметр varchar(20)`. Добавить проверку: имя длиннее 20 символов — ошибка программиста, не пользователя. Бросить `InvalidOperationException` с внятным текстом. Плюс unit-тест, который прогоняет **все** префиксы с реалистично большим идентификатором грида (например 9999999) и проверяет границу. Этот тест — страховка на будущее: он упадёт, когда кто-то добавит слишком длинный префикс.

4. **Заменить в трёх местах-потребителях** самодельную сборку имён на вызов этого метода, если такая сборка появилась. Существующие места чтения/записи личных параметров **не рефакторить** — они работают, трогать их не просили (Surgical Changes). Реестр нужен новому коду; переводить на него старый — отдельная задача, о которой достаточно упомянуть в отчёте.

## Чего НЕ делать

- Не заводить конфигурируемый список префиксов, реестр в БД, атрибуты, рефлексию по типам.
- Не переименовывать существующие префиксы и не менять формат существующих ключей — это сломает уже сохранённые настройки пользователей.
- Не добавлять в набор параметры, которые к гриду не относятся (глобальные настройки приложения, тема оформления и т.п.).

## → verify

1. Unit-тест: набор для конкретного идентификатора грида содержит ожидаемые имена и **ровно** их (сравнение множеств, не подстрок).
2. Unit-тест: у всех имён длина ≤ 20 при большом идентификаторе грида.
3. Unit-тест: наборы для двух разных гридов не пересекаются.
4. Поиск по решению подтверждает, что в новом коде нет ни одного места, где имя параметра грида собирается конкатенацией вручную.
5. `dotnet build` + `dotnet test` зелёные.
