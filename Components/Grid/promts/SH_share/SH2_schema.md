# SH2 — Схема БД: общие настройки

> Требует `SH1`. Правила — в `SH0`. **Самый ответственный этап: здесь ломается или сохраняется целостность существующих настроек.**

## Цель

Дать схеме возможность хранить наборы параметров, привязанные к общей настройке, не поломав существующее хранение личных параметров.

Изменения — в `scripts/dynamic-grid/schema.sql` (dev/test-стенд, LocalDB). Скрипт **идемпотентный**, перезапускаемый.

**Что в файле уже есть** (сверено): `ClayGridSettings`, `ClayGridColumns` с `УчаствуетВБыстромПоиске`, `ClayGridUserParams` с двухколоночным `UQ_ClayGridUserParams` и триггером `TR_ClayGridUserParams_Upsert`, seed грида №140. Ничего из перечисленного не удалять и не переписывать сверх пунктов ниже.

**Образец идемпотентности — брать из этого же файла.** Колонка быстрого поиска добавлена там двумя приёмами: `IF COL_LENGTH(...) IS NULL ALTER TABLE ... ADD` для существующих стендов и `IF NOT EXISTS ... INSERT ... ELSE UPDATE ... FROM (VALUES ...) AS c(...)` для seed. Новые изменения оформлять **тем же стилем**, свой не изобретать.

## 1. Новая таблица

```sql
IF OBJECT_ID('dbo.ClayGridUserSharedParams', 'U') IS NULL
BEGIN
    CREATE TABLE dbo.ClayGridUserSharedParams (
        КодНастройкиОбщей int IDENTITY(1,1) NOT NULL PRIMARY KEY, -- идентификатор общей настройки
        Название nvarchar(100) NOT NULL                            -- название ссылки, показывается пользователю
    );
END
GO
```

- `IDENTITY(1,1)` — нумерация реальных настроек с 1.
- Имя таблицы в стенде — `dbo.ClayGridUserSharedParams`. В приложении имя **конфигурируемо** (`SH3`), поэтому в коде хардкода имени быть не должно.
- Колонок владельца и даты не добавлять: владелец выводится из `ClayGridUserParams.КодНастройкиКлиента` (решение 1 в `SH0`).

## 1a. Служебная запись `КодНастройкиОбщей = 0`

Решение заказчика: у обычных (личных) параметров `КодНастройкиОбщей = 0`, и в таблице общих настроек для этого есть **служебная запись с идентификатором 0**. Она обязательна — без неё внешний ключ отвергнет все личные строки.

```sql
-- Служебная запись: КодНастройкиОбщей = 0 означает личные настройки пользователя,
-- не относящиеся ни к одной общей ссылке. Требуется для внешнего ключа.
IF NOT EXISTS (SELECT 1 FROM dbo.ClayGridUserSharedParams WHERE КодНастройкиОбщей = 0)
BEGIN
    SET IDENTITY_INSERT dbo.ClayGridUserSharedParams ON;
    INSERT INTO dbo.ClayGridUserSharedParams (КодНастройкиОбщей, Название)
        VALUES (0, N'(личные настройки)');
    SET IDENTITY_INSERT dbo.ClayGridUserSharedParams OFF;
END
GO
```

- Явная вставка `0` в `IDENTITY`-колонку возможна **только** при `SET IDENTITY_INSERT ON`. Без него запрос упадёт.
- `IDENTITY(1,1)` не выдаст `0` автоматически — коллизии с этой записью не будет.
- Служебная запись **не должна появляться в UI**: ни в списке общих настроек, ни в результатах поиска. Фильтр `КодНастройкиОбщей <> 0` — обязательное условие во всех запросах `SH3`, работающих со списком.

## 1b. Порядок операций в скрипте — строго такой

Нарушение порядка приведёт к падению на внешнем ключе:

1. создать `dbo.ClayGridUserSharedParams`;
2. вставить служебную запись `0`;
3. добавить колонку `КодНастройкиОбщей` в `dbo.ClayGridUserParams` (п. 2);
4. добавить внешний ключ (п. 3).

Если добавить внешний ключ раньше служебной записи, существующие строки личных параметров уже будут содержать `0`, родителя для которого ещё нет, — проверка ключа не пройдёт.

## 2. Новое поле в `ClayGridUserParams`

```sql
IF COL_LENGTH('dbo.ClayGridUserParams', 'КодНастройкиОбщей') IS NULL
    ALTER TABLE dbo.ClayGridUserParams
        ADD КодНастройкиОбщей int NOT NULL
            CONSTRAINT DF_ClayGridUserParams_КодНастройкиОбщей DEFAULT 0;
GO
```

- `0` = личная настройка пользователя (решение заказчика). `NULL` не используется — это упрощает и уникальный ключ, и триггер.
- `DEFAULT 0` заполняет уже существующие строки, то есть все текущие личные настройки автоматически получают корректное значение.
- Ограничение по умолчанию **именованное** — иначе скрипт нельзя будет сделать идемпотентным, а автосгенерированное имя невозможно предсказать.

## 3. Внешний ключ — **без** каскада

```sql
IF OBJECT_ID('dbo.FK_ClayGridUserParams_SharedParams', 'F') IS NULL
    ALTER TABLE dbo.ClayGridUserParams
        ADD CONSTRAINT FK_ClayGridUserParams_SharedParams
        FOREIGN KEY (КодНастройкиОбщей)
        REFERENCES dbo.ClayGridUserSharedParams (КодНастройкиОбщей);
GO
```

**`ON DELETE CASCADE` здесь ставить нельзя.** С появлением служебной записи `0` каскад превращается в оружие: единственный ошибочный `DELETE FROM dbo.ClayGridUserSharedParams WHERE КодНастройкиОбщей = 0` — из скрипта, из чужой чистки «мусорных» записей, из невнимательного `DELETE` без `WHERE` — молча удалит **личные настройки всех пользователей по всем гридам**. Восстановить их будет нечем.

Без каскада внешний ключ работает наоборот, как предохранитель: такой `DELETE` **упадёт**, потому что на запись `0` ссылаются строки личных параметров. Это именно то поведение, которое здесь нужно.

Цена — удаление общей настройки становится двухшаговым (сначала её строки параметров, затем сама запись). Это несколько строк кода в `SH3` против риска потери всех пользовательских настроек. Размен однозначный.

Триггер `INSTEAD OF INSERT` не удалять и в `INSTEAD OF DELETE` не превращать. Кстати, при выбранном варианте это ограничение и не мешает: SQL Server запрещает `INSTEAD OF DELETE`-триггер именно на таблице, участвующей в каскадном удалении, а каскада у нас теперь нет.

## 4. Перестройка уникального ключа

Существующий `UNIQUE (КодНастройкиКлиента, Параметр)` **обязан** быть заменён:

```sql
IF OBJECT_ID('dbo.UQ_ClayGridUserParams', 'UQ') IS NOT NULL
    ALTER TABLE dbo.ClayGridUserParams DROP CONSTRAINT UQ_ClayGridUserParams;
GO

IF OBJECT_ID('dbo.UQ_ClayGridUserParams', 'UQ') IS NULL
    ALTER TABLE dbo.ClayGridUserParams
        ADD CONSTRAINT UQ_ClayGridUserParams
        UNIQUE (КодНастройкиКлиента, Параметр, КодНастройкиОбщей);
GO
```

Почему обязательно: без `КодНастройкиОбщей` в ключе пользователь может иметь только **одну** строку на имя параметра, и второй «поделиться» по тому же гриду упадёт с нарушением уникальности.

Поскольку личные строки несут `0`, а не `NULL`, семантика ключа прозрачна: на пару (пользователь, параметр) остаётся ровно одна личная строка (`0`) плюс по одной строке на каждую общую настройку. Никаких особенностей сравнения `NULL` в `UNIQUE` учитывать не нужно.

## 5. Переписать триггер upsert

Триггер соединяет строки по ключу и **обязан** учитывать `КодНастройкиОбщей`, иначе сохранение личного параметра перезапишет значение в общей настройке — молча испортив уже разосланные ссылки.

Поскольку `NULL` в этой колонке не используется (личные строки несут `0`), сравнение — **обычное равенство**. `ISNULL`-обвязка не нужна и добавлять её не надо.

```sql
IF OBJECT_ID('dbo.TR_ClayGridUserParams_Upsert', 'TR') IS NOT NULL
    DROP TRIGGER dbo.TR_ClayGridUserParams_Upsert;
GO

CREATE TRIGGER dbo.TR_ClayGridUserParams_Upsert
ON dbo.ClayGridUserParams
INSTEAD OF INSERT
AS
BEGIN
    SET NOCOUNT ON;

    -- Обновление существующих строк. КодНастройкиОбщей входит в ключ:
    -- 0 — личная настройка пользователя, ненулевое значение — настройка общей ссылки.
    UPDATE tgt
       SET tgt.Значение = src.Значение
      FROM dbo.ClayGridUserParams AS tgt
     INNER JOIN inserted AS src
        ON tgt.КодНастройкиКлиента = src.КодНастройкиКлиента
       AND tgt.Параметр = src.Параметр
       AND tgt.КодНастройкиОбщей = src.КодНастройкиОбщей;

    -- Вставка отсутствующих строк.
    INSERT INTO dbo.ClayGridUserParams (КодНастройкиКлиента, Параметр, Значение, КодНастройкиОбщей)
    SELECT src.КодНастройкиКлиента, src.Параметр, src.Значение, src.КодНастройкиОбщей
      FROM inserted AS src
     WHERE NOT EXISTS (
               SELECT 1
                 FROM dbo.ClayGridUserParams AS tgt
                WHERE tgt.КодНастройкиКлиента = src.КодНастройкиКлиента
                  AND tgt.Параметр = src.Параметр
                  AND tgt.КодНастройкиОбщей = src.КодНастройкиОбщей
           );
END
GO
```

**Важно:** приложение обязано передавать `КодНастройкиОбщей` в каждом `INSERT` явно — `0` для личных настроек, идентификатор общей настройки для общих. Полагаться на `DEFAULT 0` в коде нельзя: колонка перечислена в списке вставки триггера, и пропуск значения в вызывающем запросе даст не то, что ожидается.

Известное ограничение, которое **уже было** в исходном триггере: если в одном `INSERT` придут две строки с одинаковым ключом, `NOT EXISTS` пропустит обе. Это не регресс от текущей задачи — **отметить в отчёте и не исправлять** (Surgical Changes). Со стороны приложения в одном `INSERT` дубликатов ключа быть не должно (`SH3`).

## 5a. Табличная функция чтения общих параметров

Приложение читает параметры по `sharedId` **только** через объект БД, имя которого приходит из `appsettings.json` (параметр `UserParamsShared`, см. `SH3`). На стенде этот объект нужно создать, иначе режим `sharedId` проверить нечем.

Контракт задан заказчиком: входной параметр `@КодНастройкиОбщей`, возвращаемые поля `Параметр varchar(50)`, `Значение nvarchar(MAX)`.

```sql
IF OBJECT_ID('dbo.ClayGridUserParamsShared', 'IF') IS NOT NULL
    DROP FUNCTION dbo.ClayGridUserParamsShared;
GO

-- Возвращает параметры общей настройки по её идентификатору.
-- Фильтрация только по КодНастройкиОбщей: настройку открывает другой пользователь,
-- поэтому условия по КодНастройкиКлиента здесь быть не должно.
-- Ноль отсекается намеренно: 0 — служебное значение личных настроек,
-- выборка по нему вернула бы личные параметры всех пользователей.
CREATE FUNCTION dbo.ClayGridUserParamsShared (@КодНастройкиОбщей int)
RETURNS TABLE
AS
RETURN
(
    SELECT CAST(Параметр AS varchar(50))    AS Параметр,  -- имя параметра грида
           CAST(Значение AS nvarchar(MAX))  AS Значение   -- сохранённое значение параметра
      FROM dbo.ClayGridUserParams
     WHERE КодНастройкиОбщей = @КодНастройкиОбщей
       AND @КодНастройкиОбщей <> 0
);
GO
```

- Тип объекта — **встроенная табличная функция** (`OBJECT_ID(..., 'IF')`). Заказчик называет объект хранимой процедурой, но описывает его как табличную функцию с возвращаемыми полями; на стенде реализуем функцию, а фактический тип объекта в целевой БД перед написанием вызова **обязательно проверить** (`SH0`, ловушка 10).
- `CAST` к типам контракта оставить явно, даже что колонки таблицы уже́ и по типу отличаются: контракт функции — публичный интерфейс, он не должен меняться вслед за типами таблицы.
- Имя объекта на стенде — `dbo.ClayGridUserParamsShared`; в приложении имя конфигурируемо, хардкода в коде быть не должно.

## Чего НЕ делать

- Не менять существующие колонки, их типы, `Параметр varchar(20)`, `Значение varchar(1000)`.
- Не добавлять индексы «на будущее», `CHECK`-констрейнты, вычисляемые колонки, колонки владельца/даты.
- Не превращать триггер в `INSTEAD OF DELETE/UPDATE` и не удалять его.
- Не писать миграцию продакшн-БД и не трогать файлы вне `scripts/dynamic-grid/schema.sql`.
