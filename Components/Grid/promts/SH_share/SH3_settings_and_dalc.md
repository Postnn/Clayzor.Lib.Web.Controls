# SH3 — Конфигурация и слой доступа к данным

> Требует `SH1`, `SH2`. Правила — в `SH0`.

## 1. Два новых параметра в `appsettings.json`

Заказчик указал два новых значения, оба задаются **в `appsettings.json`**:

| Параметр | Что задаёт | Значение по умолчанию (совпадает со стендом `SH2`) |
| --- | --- | --- |
| имя для `UserSharedParamsTable` | таблица общих настроек | `ClayGridUserSharedParams` |
| **`UserParamsShared`** | объект БД для чтения параметров по `sharedId` | `ClayGridUserParamsShared` |

- Ключи и способ доставки в код — **строго по той же схеме, которой уже заданы имена `ClayGridSettings` / `ClayGridColumns` / `ClayGridUserParams`** (установлено в `SH1`, вопрос 16). Второй механизм конфигурации не изобретать.
- Оба свойства — с `/// <summary>`, где указан ключ конфигурации. Для `UserParamsShared` в комментарии зафиксировать контракт: входной параметр `@КодНастройкиОбщей`, возвращаемые поля `Параметр varchar(50)`, `Значение nvarchar(MAX)`.
- Имена объектов в SQL-константах **не хардкодить**: подставлять из настроек тем же приёмом, каким это уже сделано для остальных объектов динамического грида.

**Внимание к безопасности:** имена таблицы и функции — единственные места, где значение попадает в текст SQL, потому что имена объектов параметром Dapper передать нельзя. Значения приходят из конфигурации приложения, не от пользователя, и это допустимо — **но** они обязаны использоваться так же, как уже используются существующие имена таблиц (тот же приём экранирования/квотирования, если он есть). Никогда не подставлять в имя объекта что-либо, полученное из URL, формы или БД.

## 1a. ⚠️ Матрица доступа к параметрам — обязательна к соблюдению

Это самый вероятный источник тихой поломки во всей задаче. Пять путей, у каждого **свой** обязательный набор условий. Ни один запрос не имеет права обращаться к `ClayGridUserParams` без условия по `КодНастройкиОбщей` и без фильтра по именам параметров текущего грида (реестр `SH4`).

| Что делаем | Источник | Обязательные условия |
| --- | --- | --- |
| Читаем текущие настройки грида (личные) | `UserParamsTable` | `КодНастройкиКлиента` = текущий пользователь **И** `КодНастройкиОбщей = 0` **И** `Параметр` входит в имена параметров текущего грида |
| Читаем список своих общих настроек | `UserParamsTable` (+ название из `UserSharedParamsTable`) | `КодНастройкиКлиента` = текущий пользователь **И** `КодНастройкиОбщей <> 0` **И** `Параметр` входит в имена параметров текущего грида, `DISTINCT` по настройке |
| Читаем параметры по `sharedId` (открывающий ссылку) | **только** `UserParamsShared(@КодНастройкиОбщей)` | фильтр по именам параметров текущего грида; **никакого** условия по `КодНастройкиКлиента`; в `UserParamsTable` за этим не ходить |
| Сохраняем текущие настройки грида | `INSERT` в `UserParamsTable` | `КодНастройкиОбщей = 0` **явно** в списке вставки |
| Сохраняем общие настройки при «поделиться» | `INSERT` в `UserParamsTable` | `КодНастройкиОбщей` = новый идентификатор **явно** в списке вставки |

### Почему это критично

1. **Существующие чтения сломаются молча.** Сейчас личные параметры читаются запросом вида `... WHERE КодНастройкиКлиента = @user` — без упоминания `КодНастройкиОбщей`, которой раньше не существовало. После `SH2` в той же таблице появляются строки общих настроек, и **тот же запрос вернёт их вместе с личными**: то, чем пользователь когда-то поделился, подмешается в его собственные настройки. Внешне это выглядит как «настройки грида сами меняются» — диагностировать крайне тяжело.

2. **Существующие записи сломаются тоже.** Триггер перечисляет `КодНастройкиОбщей` в списке вставки (`SH2`, п. 5), поэтому пропуск значения в вызывающем запросе даёт не то, что ожидается. Полагаться на `DEFAULT 0` нельзя — значение передаётся явно в **обоих** путях записи.

### Что сделать

1. Найти **все** запросы, читающие и пишущие `ClayGridUserParams` (установлено в `SH1`, вопрос 3).
2. Привести каждый в соответствие таблице выше.
3. Проверить поиском по решению, что не осталось ни одного обращения к таблице без условия по `КодНастройкиОбщей`.
4. Фильтр по именам параметров брать из реестра `SH4`, а не собирать список руками в каждом запросе.

Это правка существующего кода, и она **не** нарушает принцип Surgical Changes: без неё новая колонка ломает старое поведение. В отчёте перечислить все затронутые запросы.

## 2. Операции доступа к данным

Добавить в слой работы с динамическим гридом (рядом с чтением/записью пользовательских параметров, найденным в `SH1`) следующие операции. Каждая — с `///` XML-doc, SQL-константы с `--` комментариями на каждую колонку.

1. **Создать общую настройку** → возвращает новый `КодНастройкиОбщей`:

```sql
-- Создание общей настройки. SCOPE_IDENTITY() возвращает идентификатор,
-- выданный в текущей области видимости, — безопаснее @@IDENTITY при наличии триггеров.
INSERT INTO <таблица> (Название) VALUES (@Название);
SELECT CAST(SCOPE_IDENTITY() AS int);
```

2. **Сохранить набор параметров под общей настройкой** — пачка строк в `ClayGridUserParams` с **явно** заполненным `КодНастройкиОбщей` (новый идентификатор). **Только `INSERT`** (upsert делает триггер). `UPDATE`, `MERGE`, «`DELETE` + `INSERT`» запрещены. В одном `INSERT` не должно быть двух строк с одинаковым `Параметр` — см. известное ограничение триггера в `SH2`. Имена параметров берутся из реестра `SH4`, посторонних имён в наборе быть не должно.

3. **Список общих настроек текущего грида** — те записи, на которые ссылаются строки `ClayGridUserParams` с именами параметров текущего грида и `КодНастройкиКлиента` = текущий пользователь (решение 2 в `SH0` — только свои ссылки). Возвращать `КодНастройкиОбщей` + `Название`, отсортированные по названию.
   - `DISTINCT` обязателен: на одну общую настройку приходится много строк параметров.
   - **Обязательное условие `КодНастройкиОбщей <> 0`** — служебная запись не должна попадать в список (`SH2`, п. 1a). Без этого условия пользователь увидит в списке «(личные настройки)» с кнопкой «удалить».

4. **Есть ли общие настройки у текущего грида** — лёгкая проверка для видимости кнопки списка. Возвращать признак/количество, а не полный список. Условие `КодНастройкиОбщей <> 0` обязательно и здесь: иначе кнопка списка будет видна всегда, ведь личные параметры есть у каждого.

5. **Переименовать общую настройку** — `UPDATE` только поля `Название` по `КодНастройкиОбщей`, с условием `КодНастройкиОбщей <> 0`.

6. **Удалить общую настройку** — **два шага**, каскада в схеме нет намеренно (`SH2`, п. 3):
   1. `DELETE` строк `ClayGridUserParams` с этим `КодНастройкиОбщей`;
   2. `DELETE` самой записи из таблицы общих настроек.

   Оба шага — атомарно (см. п. 3 этого этапа). Порядок обратный нельзя: внешний ключ не даст удалить родителя, пока есть дочерние строки.

   **Защита от `0` обязательна.** В оба запроса добавить условие `КодНастройкиОбщей <> 0`, а в C#-коде — предварительную проверку с `InvalidOperationException` при попытке удалить `0`. Внешний ключ здесь предохранитель последней надежды: без него удаление служебной записи снесло бы личные настройки всех пользователей. Не полагаться только на него — ошибка должна отсекаться раньше и с внятным текстом.

7. **Прочитать параметры по `sharedId`** — **только через объект `UserParamsShared`**, имя берётся из конфигурации (п. 1). Прямой запрос к `ClayGridUserParams` по `КодНастройкиОбщей` в новом коде запрещён.

```sql
-- Чтение параметров общей настройки. Имя объекта подставляется из конфигурации
-- (ключ UserParamsShared). Фильтрация по пользователю отсутствует намеренно:
-- ссылку открывает другой пользователь.
SELECT Параметр,  -- имя параметра грида
       Значение   -- сохранённое значение параметра
  FROM <UserParamsShared>(@КодНастройкиОбщей)
```

   - **Форма вызова зависит от фактического типа объекта.** Приведённый вариант — для табличной функции. Если в целевой БД это хранимая процедура, вызов будет через `EXEC` с чтением результирующего набора. **Проверить тип объекта до написания кода** (`SH0`, ловушка 10) и при расхождении с описанием заказчика спросить, а не выбирать наугад.
   - Модель результата: `Параметр` — `string` до 50 символов, `Значение` — `string` без ограничения длины (`nvarchar(MAX)`). Не переиспользовать модель личных параметров, если она завязана на `varchar(20)`/`varchar(1000)`: контракт объекта шире, и обрезка приведёт к потере значений.
   - **`0` и отрицательные значения в этот метод не передавать.** Ноль — служебное значение личных настроек; выборка по нему вернула бы личные параметры всех пользователей. Отсечь в C# до вызова, с `InvalidOperationException`. Функция стенда дополнительно возвращает пустой набор для нуля (`SH2`, п. 5a), но полагаться только на неё нельзя — в целевой БД объект чужой.
   - Пустой результат — **не** ошибка уровня данных: это штатный сигнал «настройки не найдены», который обрабатывается в `SH8`. Исключение здесь не бросать.

## 3. Атомарность создания

Создание общей настройки — это `INSERT` в родительскую таблицу плюс пачка строк параметров. Если второй шаг упадёт, останется запись без параметров: она попадёт в список, но ничего не восстановит.

- Если `DbManager` поддерживает транзакции (`SH1`, вопрос 6) — выполнять оба шага в **одной транзакции**.
- Если не поддерживает — **не добавлять** транзакционную инфраструктуру в DALC ради этой задачи (Simplicity First). Сделать компенсацию: при ошибке сохранения параметров удалить только что созданную родительскую запись, и лишь затем дать ошибке подняться. Логику компенсации описать в `///`-комментарии.
- Выбор и причину указать в отчёте.

## 4. Ограничения полей

- `Название` — `nvarchar(100)`. Перед сохранением: `Trim()`, проверка на непустоту, проверка длины ≤ 100. Обрезать молча нельзя — валидация на стороне UI (`SH5`), но и операция данных обязана защититься.
- `Параметр` — `varchar(20)`, `Значение` — `varchar(1000)`. Правила те же, что для личных параметров; ничего нового не вводить.

## Чего НЕ делать

- Не создавать репозиторий/абстракцию «на будущее», если в проекте параметры читаются и пишутся простыми методами — следовать существующему стилю.
- Не добавлять кэш общих настроек: заказчик требует перезагружать список из БД при каждом раскрытии.
- Не добавлять `async void`, не запускать операции fire-and-forget.
- Не подавлять `SqlException` — обработка ошибок БД в проекте централизована (`ISqlErrorHandler` → `ClayErrorBar`).

1. Оба ключа появились в `appsettings.json`, значения доходят до кода; смена значения на несуществующее имя объекта приводит к штатной ошибке БД, а не к тихому неверному поведению.
2. Unit-тесты на валидацию `Название`: пусто, только пробелы, 100 символов, 101 символ.
3. Создание возвращает новый положительный идентификатор; два вызова подряд дают разные идентификаторы.

5. Проверка наличия общих настроек у грида, где их нет, возвращает «нет» — несмотря на существующие личные параметры.

13. Чтение личных параметров не возвращает строки общих настроек (создать общую настройку и убедиться, что личный набор не изменился).
