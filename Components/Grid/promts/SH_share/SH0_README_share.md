# SH0 — «Поделиться» настройками динамического ClayGrid (индекс)

> **Читать первым. Этапы выполнять строго по порядку, каждый — отдельный файл.**
> Стиль исполнения — по корневому `AGENTS.md`: **Think Before Coding → Simplicity First → Surgical Changes → Goal-Driven Execution.**

## Глобальные правила (действуют на все этапы)

- Пользовательский текст и Razor-разметка — **на русском**. Каждый public/protected член — с `/// <summary>`.
- Каждая SQL-константа — `///` XML-doc + `--` комментарий на **каждую** колонку.
- **SQL Server 2008 R2** — только доступный там синтаксис. Никаких `OFFSET/FETCH`, `TRY_CONVERT`, `MERGE` в новом коде, `STRING_SPLIT`, `IIF`, `CONCAT_WS`, `THROW`.
- **Запрещено** подставлять значения в SQL-строку. Только Dapper-параметры. При сыром SQL — `commandType: CommandType.Text`.
- Имена колонок БД — точные русские, не переводить; новые имена — ровно один раз в `ColumnNames.cs`.
- Готовые компоненты проекта обязательны: `ClayButton` (кнопки с тултипом), `ClayMenu` (выпадающие меню), `ConfirmDialog` (подтверждения), `ClayCheckbox`. Форма ввода — `Variant.Outlined`. Все диалоги — draggable.
- Git — **только по прямому указанию**. `AGENTS.md` и `docs/` не обновлять.
- Неоднозначность → **вопрос через question tool**, с озвученными предположениями.
- После каждого этапа: `dotnet build Clayzor.sln` + `dotnet test tests\Clayzor.Lib.Web.Controls.Tests`. На красной сборке дальше не идти.

## Задача

Пользователь настроил грид (группировка, фильтры, сортировка, порядок и видимость колонок, быстрый поиск и прочее, что сохраняется в `UserParamsTable`) и одной кнопкой получает в буфер обмена ссылку для коллеги.

**По кнопке «Поделиться»:**
1. Диалог ввода названия ссылки, по умолчанию — название грида.
2. Кнопка «Получить ссылку»:
   1. новая запись в `UserSharedParamsTable` (имя таблицы — новый параметр в `appsettings.json`); поля `КодНастройкиОбщей int` (PK + автоинкремент), `Название nvarchar(100)`;
   2. параметры текущего грида сохраняются с новым `КодНастройкиОбщей`; в `UserParamsTable` добавлено поле `КодНастройкиОбщей` — FK на `UserSharedParamsTable`;
   3. ссылка = текущий URL грида, очищенный от всех параметров кроме идентификатора грида, плюс `sharedId = КодНастройкиОбщей`;
   4. ссылка копируется в буфер обмена, показывается временное всплывающее сообщение об успешном копировании, диалог закрывается.

**Рядом с кнопкой** — раскрывающийся список общих настроек текущего грида, показывается **только если они есть**. Список **каждый раз перезагружается из БД**. У каждого элемента кнопки в стиле приложения: **редактировать** (диалог переименования), **удалить** (с подтверждением), **перейти** (новое окно браузера), **поделиться** (копировать ссылку). Любая из кнопок закрывает список.

**Грид со `sharedId` в строке запроса:**
1. Параметры общей настройки читаются **исключительно через объект БД, имя которого задано в `appsettings.json` в параметре `UserParamsShared`** (табличная функция, входной параметр `@КодНастройкиОбщей`, возвращает `Параметр varchar(50)`, `Значение nvarchar(MAX)`). Напрямую к `UserParamsTable` за общими настройками не обращаться.
2. После загрузки проверяется, что это параметры **текущего** грида.
3. **Если параметров нет или это чужие параметры — выдать сообщение об ошибке и грид не загружать.** Никакого молчаливого перехода на личные настройки.
4. Если `sharedId` не передан или равен `0` — считаем, что параметр не передан, работаем стандартно.
5. **Ключевое:** в режиме `sharedId` никакие изменения параметров **не сохраняются** для текущего пользователя. Грид со `sharedId` всегда открывается с параметрами того, кто ссылку дал.

## Проверенные факты схемы (из `scripts/dynamic-grid/schema.sql` — не выдумывать заново)

```
dbo.ClayGridSettings:    КодЗапроса int PK, Запрос varchar(50), Пиктограмма varchar(50),
                         [SQL] varchar(4000), ID varchar(50), IDName varchar(50),
                         ФормаРедактирования varchar(100), ФормаНового varchar(100),
                         SQLDelete varchar(300)

dbo.ClayGridColumns:     КодКолонки int PK, КодЗапроса int, Колонка varchar(50),
                         ЗаголовокКолонки varchar(50), КлючURL varchar(50), Порядок int,
                         Формат varchar(2000), Тип int, УчаствуетВБыстромПоиске tinyint

dbo.ClayGridUserParams:  КодНастройкиКлиента int, Параметр varchar(20), Значение varchar(1000),
                         UNIQUE (КодНастройкиКлиента, Параметр)
                         + триггер TR_ClayGridUserParams_Upsert (INSTEAD OF INSERT) делает upsert
                           → приложение отправляет ТОЛЬКО INSERT
```

Название грида для дефолтного имени ссылки — `ClayGridSettings.Запрос` (`varchar(50)`).

**Состояние схемы на момент постановки задачи** (сверено с `scripts/dynamic-grid/schema.sql`):

- Быстрый поиск **уже реализован**: `УчаствуетВБыстромПоиске tinyint NULL` есть в `CREATE TABLE`, в идемпотентном `ALTER` и в seed грида №140. Значит соответствующий пользовательский параметр — **реальный**, а не гипотетический: он обязан войти в реестр (`SH4`), копироваться при «поделиться» (`SH5`) и подчиняться запрету сохранения (`SH8`).
- Под «поделиться» в схеме **ничего нет**: колонки `КодНастройкиОбщей` не существует, таблицы общих настроек нет, объекта `UserParamsShared` нет, `UNIQUE` по-прежнему двухколоночный `(КодНастройкиКлиента, Параметр)`, триггер соединяет строки только по этим двум колонкам. Этап `SH2` актуален целиком.
- `Параметр varchar(20)` и `Значение varchar(1000)` не менялись.

**Ограничение источников.** Каталог `src/` подключён как git-submodule, поэтому при составлении промта C#-код был недоступен: сверены только корневой `AGENTS.md` и `scripts/dynamic-grid/schema.sql`. Все имена классов, методов, свойств и параметров ниже — **ориентиры, а не факты**. Устанавливает их агент на этапе `SH1` по рабочей копии. При расхождении промта с кодом верить коду и сообщить о расхождении в отчёте.

**Решение заказчика по новому полю:** `КодНастройкиОбщей` входит в уникальный ключ; у обычных (личных) параметров значение равно **`0`**, `NULL` не используется. В `UserSharedParamsTable` для этого существует **служебная запись с `КодНастройкиОбщей = 0`**, благодаря которой внешний ключ выполняется и для личных строк.

**Матрица доступа к параметрам** (уточнение заказчика, канонический ориентир; развёрнуто — `SH3`, п. 1a):

| Что делаем | Источник | Обязательные условия |
| --- | --- | --- |
| Читаем текущие настройки грида | `UserParamsTable` | текущий пользователь + `КодНастройкиОбщей = 0` + имена параметров текущего грида |
| Читаем список своих общих настроек | `UserParamsTable` | текущий пользователь + `КодНастройкиОбщей <> 0` + имена параметров текущего грида |
| Читаем параметры по `sharedId` | **только** `UserParamsShared` | имена параметров текущего грида; без условия по пользователю |
| Сохраняем текущие настройки | `INSERT` в `UserParamsTable` | `КодНастройкиОбщей = 0` явно |
| Сохраняем общие настройки | `INSERT` в `UserParamsTable` | новый `КодНастройкиОбщей` явно |

Оба пути записи меняются из-за появления колонки: значение передаётся **явно**, полагаться на `DEFAULT` нельзя.

**Чтение общих параметров — только через объект БД.** Имя объекта задаётся в `appsettings.json`, параметр **`UserParamsShared`**. Контракт:

```
<UserParamsShared>(@КодНастройкиОбщей int)
    → Параметр  varchar(50)
      Значение  nvarchar(MAX)
```

Обратите внимание: типы в контракте **шире**, чем в таблице (`Параметр varchar(20)`, `Значение varchar(1000)`). Это нормально для чтения, но означает, что функция может вернуть имя параметра, которое в таблицу в принципе не поместилось бы, — такое имя заведомо не принадлежит текущему гриду и обрабатывается проверкой соответствия (`SH8`).

## ⚠️ Ловушки — прочитать до кода

**1. Уникальный ключ и триггер обязаны быть перестроены. Это блокер, а не деталь.**
Исходный `UNIQUE (КодНастройкиКлиента, Параметр)` разрешает пользователю **одну** строку на имя параметра — второй «поделиться» по тому же гриду упал бы. Плюс триггер `INSTEAD OF INSERT` соединяет строки только по (`КодНастройкиКлиента`, `Параметр`), то есть **сохранение личной настройки перезаписало бы значение в общей**, молча испортив уже отправленные коллегам ссылки. По решению заказчика `КодНастройкиОбщей` входит в уникальный ключ; триггер обязан учитывать его в тех же условиях соединения — этап `SH2`. Пропустить этот этап или сделать его «потом» нельзя.

**2. Существующие запросы личных параметров начнут видеть общие.** Сейчас они читают таблицу без условия по `КодНастройкиОбщей`, которой не существовало. После появления колонки те же запросы вернут и строки общих настроек — настройки, которыми пользователь поделился, подмешаются в его собственные. Внешне это «настройки грида сами меняются», диагностируется тяжело. Во все чтения личных параметров добавить `AND КодНастройкиОбщей = 0`, во все записи — явный `0`. Детали — `SH3`, п. 1a.

**3. `Параметр varchar(20)`** — 20 символов. Все ключи параметров (включая `QuickSearchParamPrefix` из задачи быстрого поиска) обязаны влезать. Ключи из имён колонок не строить.

**4. Блокировка сохранения — в одной точке, а не «где вспомнил».** Параметры пишут многие места: диалог настройки колонок, группировка, фильтры, сортировка, размер страницы, быстрый поиск. Расставлять `if (sharedMode)` по всем вызывающим — гарантированная дырка. Запрет — **один choke point** в слое сохранения параметров. Детали — `SH8`.

**5. Буфер обмена может быть недоступен.** `navigator.clipboard` работает только в secure context: `https` или `localhost`. На интранет-хосте по обычному `http` метод отсутствует/бросает исключение, и «ссылка скопирована» окажется ложью. Нужен fallback — `SH6`.

**6. `ON DELETE CASCADE` ставить нельзя.** Личные строки несут `КодНастройкиОбщей = 0`, и в таблице общих настроек для этого есть служебная запись `0`. С каскадом единственный ошибочный `DELETE ... WHERE КодНастройкиОбщей = 0` молча снесёт **личные настройки всех пользователей по всем гридам**. Без каскада внешний ключ отработает предохранителем и такой `DELETE` просто упадёт. Удаление настройки делается двумя шагами. Детали — `SH2` п. 3, `SH3` операция 6.

**8. Служебная запись `0` не должна попадать в UI.** Она лежит в той же таблице, что и настоящие ссылки. Без условия `КодНастройкиОбщей <> 0` пользователь увидит в списке «(личные настройки)» с кнопкой «удалить», а кнопка списка будет видна всегда — ведь личные параметры есть у каждого. Условие обязательно во всех запросах списка и проверки наличия.

**9. Вставка записи `0` требует `SET IDENTITY_INSERT`.** Колонка — `IDENTITY`, явная вставка нуля без этого переключателя падает.

**10. Тип объекта `UserParamsShared` определяет форму вызова.** Заказчик называет его хранимой процедурой, но описывает как **табличную функцию** с возвращаемыми полями. Это разные объекты и разный вызов: функция — `SELECT ... FROM <имя>(@КодНастройкиОбщей)`, процедура — `EXEC`. **Проверить фактический тип объекта в целевой БД до написания вызова** и при расхождении с описанием спросить, а не выбирать наугад.

**11. Вызывать объект с `КодНастройкиОбщей = 0` нельзя.** Ноль — служебное значение личных настроек; выборка по нему вернула бы личные параметры **всех** пользователей. Ноль отсекается в приложении до вызова (`SH8`), и функция стенда дополнительно возвращает пустой набор для нуля (`SH2`).

**12. За общими параметрами больше не ходят напрямую в таблицу.** Единственный путь чтения по `sharedId` — объект `UserParamsShared`. Запрос вида `SELECT ... FROM ClayGridUserParams WHERE КодНастройкиОбщей = @sharedId` в новом коде появляться не должен.

**7. Не сохранять личные параметры перед формированием ссылки в режиме `sharedId`.** Иначе первый же «поделиться» из открытой по ссылке настройки запишет чужие параметры текущему пользователю — прямое нарушение п.4 задачи.

## Принятые решения (улучшения и трактовки; при несогласии — поправьте до старта)

1. **Владелец ссылки отдельной колонкой не хранится.** Он выводится из `UserParamsTable.КодНастройкиКлиента` у строк с этим `КодНастройкиОбщей`. Схему, заданную заказчиком (`КодНастройкиОбщей`, `Название`), менять не нужно.
2. **Список показывает только свои ссылки** — у элементов есть «редактировать» и «удалить», давать их на чужие настройки нельзя.
3. **Если общих параметров нет или они чужие — сообщение об ошибке, грид не загружается** (требование заказчика). Данные не запрашиваются, подмена личными настройками не выполняется.
4. **Несуществующий, нечисловой или отрицательный `sharedId`** попадает в ту же ветку: сообщение об ошибке, грид не загружается. Исключение — только отсутствие параметра и `sharedId = 0`: это штатная работа.
4a. **Улучшение (подтвердите):** в сообщении об ошибке даю кнопку «Открыть без общих настроек», которая переходит на тот же грид с очищенным `sharedId`. Иначе пользователь по мёртвой ссылке оказывается в тупике и вынужден править URL руками. Кнопка не противоречит требованию «грид не загружать»: грид со `sharedId` не загружается, а уйти с него пользователь решает сам.
5. **Имена колонок внутри значений**, которых больше нет в `ClayGridColumns`, игнорируются молча — как и в личных настройках. Несоответствием считается расхождение **имён параметров**, а не их содержимого.
6. **Из грида, открытого по `sharedId`, «поделиться» разрешено** — создаётся новая общая настройка, личные параметры при этом не пишутся.
7. **Ссылка формируется из текущего состояния грида**, а не из того, что уже лежит в `UserParamsTable`: пользователь делится тем, что видит на экране.

## Definition of Done

- [ ] Второй и третий «поделиться» по одному гриду работают без нарушения `UNIQUE`.
- [ ] Сохранение личных настроек **не меняет** значения ранее созданных общих настроек.
- [ ] Ссылка содержит только идентификатор грида и `sharedId`; повторное «поделиться» не плодит дублей `sharedId` в URL.
- [ ] Копирование в буфер подтверждается всплывающим сообщением; при недоступном буфере пользователь всё равно получает ссылку.
- [ ] Список общих настроек виден только при их наличии, перезагружается из БД при каждом раскрытии, закрывается по любой кнопке.
- [ ] Удаление снимает и запись, и её параметры; подтверждение обязательно.
- [ ] Грид по `sharedId` применяет чужие параметры и **ничего** не сохраняет: ни колонки, ни группировку, ни фильтры, ни сортировку, ни быстрый поиск, ни размер страницы.
- [ ] `sharedId` отсутствует / `= 0` → стандартная работа без сообщений.
- [ ] Общих параметров нет / параметры чужого грида / битый `sharedId` → сообщение об ошибке, **грид не загружен**, запрос данных не выполнялся.
- [ ] Общие параметры читаются только через объект `UserParamsShared`; прямых запросов к таблице по `sharedId` в новом коде нет.
- [ ] Статические гриды и остальные функции динамического не изменились.
- [ ] `dotnet build` + `dotnet test` зелёные, новые public-члены документированы, диф минимален.
