# SH6 — Формирование ссылки и копирование в буфер обмена

> Требует `SH1` (вопросы 8–10, 12, 14). Правила — в `SH0`.

## Часть 1. Формирование ссылки

Требование: **текущий URL грида, очищенный от всех параметров кроме идентификатора грида, плюс `sharedId`.**

### Правила

1. Идентификатор грида сохраняется **в том виде, в котором он есть сейчас** — часть маршрута или query-параметр (установлено в `SH1`, вопрос 8). Не переносить его из маршрута в query и наоборот.
2. Удаляются **все остальные** query-параметры: фильтры (ключи из `ClayGridColumns.КлючURL`), номер страницы, сортировка, любые прочие. Подход — «белый список»: оставить идентификатор грида, выбросить всё остальное. Список удаляемых параметров не перечислять — он будет расти и его забудут дополнить.
3. Фрагмент (`#...`) отбрасывается.
4. Добавляется `sharedId=<КодНастройкиОбщей>`.
5. **Если `sharedId` уже присутствует** в текущем URL (грид открыт по чужой ссылке и пользователь делится заново) — он **заменяется**, а не добавляется вторым. Два `sharedId` в одном URL — типичная ошибка ручной склейки строк.
6. Ссылка **абсолютная**, с той же схемой и хостом, что у текущей страницы: коллега получает готовый к вставке в браузер адрес. Относительный путь не годится.
7. Сборка — через `NavigationManager.Uri`/`BaseUri` и штатные средства работы с query (`Microsoft.AspNetCore.WebUtilities.QueryHelpers`, если он уже используется в проекте — `SH1`, вопрос 9). **Склейка строк через `+` и `Replace` запрещена.** Значение обязано быть корректно URL-кодировано.

### Ловушка: хост за прокси

Если приложение работает за реверс-прокси, `NavigationManager.BaseUri` может содержать внутреннее имя хоста, и ссылка окажется нерабочей у коллеги. Проверить на стенде и **отметить в отчёте**: если хост в сгенерированной ссылке отличается от того, что видит пользователь в адресной строке, это нужно решать настройкой forwarded headers на уровне приложения — **отдельной задачей**, не самостоятельными правками в рамках этой.

## Часть 2. Копирование в буфер обмена

### Ловушка (главная в этапе)

`navigator.clipboard` доступен только в **secure context**: `https` либо `localhost`. Дев-адрес `http://localhost:5010` — secure context, всё сработает. На интранет-хосте по обычному `http` объекта `clipboard` **не существует**, и наивный код выдаст «ссылка скопирована в буфер обмена», не скопировав ничего. Пользователь отправит коллеге пустоту.

Поэтому копирование обязано **сообщать о результате**, а не выполняться «вслепую».

### Что сделать

1. JS-хелпер в существующем скрипте RCL либо новый файл `_content/Clayzor.Lib.Web.Controls/js/clayShare.js`. Новый файл требует регистрации в `App.razor` рядом с остальными скриптами Clayzor — если можно обойтись существующим файлом, лишний не создавать.

2. Хелпер возвращает `true`/`false`, никогда не бросает наружу:
   - основной путь — `navigator.clipboard.writeText(text)`;
   - fallback при отсутствии `navigator.clipboard` или отказе — временный `textarea` + `document.execCommand('copy')` + удаление элемента;
   - обе ветки в `try/catch`, результат — булево.

3. C#-сторона:
   - `true` → всплывающее временное сообщение (Snackbar, механизм из `SH1`, вопрос 12) на русском: «Ссылка скопирована в буфер обмена». Диалог закрывается (`SH5`).
   - `false` → **не** сообщать об успехе. Показать ссылку пользователю в виде, пригодном для ручного копирования: поле только для чтения с выделенным текстом и понятной подписью на русском о том, что скопировать автоматически не удалось. Диалог в этом случае остаётся открытым.

4. Один и тот же хелпер используют и `SH5` (создание), и `SH7` (кнопка «поделиться» у элемента списка). Второй реализации быть не должно.

## Часть 3. Открытие в новом окне (для `SH7`)

Кнопка «перейти» открывает ссылку в новом окне браузера. Использовать **тот же** механизм, что уже принят в проекте для открытия внешних ссылок (`SH1`, вопрос 10): анкор с `Target="_blank"` и `rel="noopener"`, либо существующий JS-хелпер. Новый способ не изобретать.

**Запрещено:** `NavigationManager.NavigateTo(url)` без нового окна — это заменит текущую страницу и пользователь потеряет своё состояние грида.

## Чего НЕ делать

- Не сокращать ссылку, не кодировать состояние в сам URL, не добавлять QR-код.
- Не добавлять в ссылку идентификатор пользователя, токены, срок действия.
- Не логировать сформированные ссылки.
- Не пытаться «починить» хост за прокси в рамках этой задачи.

## → verify

1. Ссылка построена из URL с кучей мусорных query-параметров и фрагментом → в результате только идентификатор грида и `sharedId`.
2. Ссылка построена из URL, **уже содержащего** `sharedId=5` → в результате ровно один `sharedId`, с новым значением.
3. Ссылка абсолютная, открывается в чистом браузере (вставить вручную) и приводит на тот же грид.
4. Значение в query корректно закодировано; специальные символы в остальных параметрах не влияют на результат.
5. Unit-тесты на сборку URL — все четыре пункта выше, на чистом методе без браузера.
6. Копирование на `localhost` → успех, Snackbar появился.
7. Имитация недоступного буфера (в консоли браузера: `delete navigator.clipboard` перед нажатием, либо тест на `http`-хосте) → ложного сообщения об успехе нет, ссылка показана для ручного копирования, диалог не закрылся.
8. Кнопка «перейти» открывает **новое** окно/вкладку, исходная страница и состояние грида целы.
9. `dotnet build` + `dotnet test` зелёные.
