# SH8 — Режим `sharedId`: загрузка чужих настроек и запрет сохранения

> Требует `SH3`, `SH4`. Правила — в `SH0`. **Самый ответственный этап по последствиям: ошибка здесь молча портит личные настройки пользователей.**

## 1. Разбор параметра

Грид при инициализации читает `sharedId` из строки запроса.

| Что пришло | Трактовка |
| --- | --- |
| параметра нет | обычный режим, личные настройки |
| `sharedId = 0` | обычный режим (прямое требование заказчика) |
| пусто, не число, отрицательное | **ошибка** → сообщение, грид не загружается |
| число, объект вернул пустой набор | **ошибка** → сообщение, грид не загружается |
| число, набор непуст | переход к проверке имён (п. 2) |

Разбор — `int.TryParse` с `CultureInfo.InvariantCulture`. Никаких `Convert.ToInt32` на пользовательском вводе.

Загрузка параметров — **только через объект `UserParamsShared`** (`SH3`, операция 7). Прямого запроса к `ClayGridUserParams` по `КодНастройкиОбщей` здесь быть не должно.

## 2. Проверка соответствия имён параметров

Сравниваются **имена** параметров, возвращённые объектом `UserParamsShared`, с набором имён параметров текущего грида из реестра `SH4`.

Правило (зафиксировать в `///`-комментарии):

- **Соответствуют** — если каждое имя параметра из набора входит в набор имён текущего грида. Набор может содержать **подмножество** (пользователь не менял часть настроек — их там нет), это норма.
- **Не соответствуют** — если есть хотя бы одно имя, которого текущий грид не знает. Это значит, что ссылка сделана для другого грида либо формат ключей менялся.
- **Решено заказчиком, обсуждению не подлежит: проверка строгая, не пересечение.** Смешанный набор (несколько знакомых имён плюс одно незнакомое) — это **чужие параметры**: сообщение об ошибке, грид не загружается. Отбрасывать незнакомые имена и применять остальные **запрещено**, даже если кажется, что так «полезнее для пользователя». Частично применённые чужие настройки дают состояние, которого не было ни у кого, и молча вводят пользователя в заблуждение.
- Контракт объекта отдаёт `Параметр` как `varchar(50)`, тогда как в таблице поле — `varchar(20)`. Имя длиннее 20 символов заведомо не принадлежит ни одному гриду (реестр `SH4` это гарантирует) и потому попадает в ветку несоответствия автоматически. Отдельной проверки длины не добавлять.
- **Имена колонок внутри значений** параметров не проверяются: колонок могло не остаться, такие значения игнорируются молча — точно так же, как в личных настройках (решение 5 в `SH0`). Несоответствие — это про **имена параметров**, не про их содержимое.

Реализовать **чистым методом** (на вход два набора строк, на выход результат проверки), чтобы покрыть тестами без БД. Сравнение имён — `OrdinalIgnoreCase`.

## 3. Сообщение об ошибке и отказ от загрузки грида

Требование заказчика: **если параметров нет или это чужие параметры — выдать сообщение об ошибке, грид не загружать.**

- **Данные не запрашиваются вообще.** Запрос строк грида не выполняется, `TotalCount` не считается, панель инструментов и таблица не рендерятся. Отказ происходит **до** загрузки данных, а не после.
- **Подмена личными настройками не выполняется.** Никакого «покажем грид как обычно» — это прямо противоречит требованию.
- Сообщение — на русском, объясняет суть без технических подробностей: ссылка недействительна либо относится к другому набору настроек, поэтому настройки не применены и таблица не загружена. Идентификатор `sharedId` в тексте показать уместно — он помогает при обращении к коллеге, давшему ссылку.
- Это **не** ошибка БД: `ClayErrorBar` и `ISqlErrorHandler` не задействовать. Использовать штатный способ проекта для нефатальных сообщений на месте содержимого страницы (`MudAlert` с `Severity.Warning`/`Error` в области грида — выбрать по образцу существующих сообщений и указать выбор в отчёте).
- Сообщение выводится **один раз на открытие**, не на каждый рендер.
- **Кнопка «Открыть без общих настроек»** (решение 4a в `SH0`, помечено «подтвердите»): переходит на тот же грид с очищенным `sharedId` — используется код очистки URL из `SH6`. Без неё пользователь по мёртвой ссылке оказывается в тупике и правит адрес руками. Переход выполняется с полной перезагрузкой, чтобы состояние компонента гарантированно сбросилось.
- Режим `sharedId` при этом **не активируется**: пользователь ничего не увидел и ничего не менял, а после перехода по кнопке работает со своим гридом, и сохранение его настроек функционирует как обычно.

## 3a. Ловушка: отказ должен наступать до запроса данных

Загрузка данных по конвенции идёт в `OnAfterRenderAsync(bool firstRender)`. Разбор `sharedId`, вызов объекта `UserParamsShared` и проверка имён обязаны выполниться **раньше** и, при неуспехе, полностью отменить загрузку. Типичная ошибка — сначала загрузить данные с настройками по умолчанию, а потом дорисовать сообщение: пользователь на мгновение увидит грид, а лишний запрос уйдёт в БД.

## 4. Применение чужих параметров

- Значения применяются **тем же кодом десериализации**, которым читаются личные параметры. Второго парсера быть не должно.
- Применить нужно всё, что пришло: группировка, фильтры, сортировка, порядок и видимость колонок, быстрый поиск, размер страницы.
- Параметры, которых в наборе нет, остаются в значении по умолчанию — **не** подмешивать личные настройки пользователя к чужому набору. Смесь двух источников даст состояние, которого не было ни у кого.
- Загрузка данных — по конвенции в `OnAfterRenderAsync(bool firstRender)` с `if (firstRender)`, не в `OnInitializedAsync`.

### Ограничения грида важнее чужих значений

Быстрый поиск уже реализован и имеет **жёсткие ограничения, не зависящие от пользователя**: если колонки `УчаствуетВБыстромПоиске` физически нет в `ColumnsTable`, если все её значения `0`/`NULL`, либо если колонка недопустима по типу (справочник, вычисляемое поле) — участие в поиске невозможно, а поле поиска скрыто.

Чужой набор параметров **не может это переопределить**. Порядок такой: сначала применяется значение из общей настройки, затем к нему применяются ограничения грида. Если после ограничений набор колонок поиска пуст — поле поиска скрыто, как и в обычном режиме.

Почему это важно: у автора ссылки грид мог быть настроен раньше, чем администратор убрал колонку из быстрого поиска или изменил её тип. Применив чужое значение «как есть», грид покажет поле поиска, которое обязано быть скрытым, и поиск уйдёт в SQL по колонке, по которой искать нельзя.

То же правило распространяется на любые другие настройки, у которых есть ограничения на стороне определения грида: **чужое значение — это пожелание, ограничение грида — закон.** Ни одно ограничение не обходится ради того, чтобы «точнее воспроизвести» чужой вид.

## 5. Запрет сохранения — главное требование этапа

> В режиме `sharedId` никакие изменения параметров не сохраняются для текущего пользователя.

### Как делать

Запрет реализуется **в единственной точке — в слое сохранения пользовательских параметров** (найден в `SH1`, вопрос 3). Флаг режима передаётся туда, и метод сохранения при активном режиме `sharedId` **не выполняет запрос** и возвращает управление без ошибки.

### Как НЕ делать

Расставлять `if (sharedMode) return;` по вызывающим местам — диалог настройки колонок, обработчики группировки, фильтрации, сортировки, смены размера страницы, применения быстрого поиска. Мест много, они добавляются со временем, и одно забытое означает, что чужие настройки затрут личные настройки пользователя без всякого признака на экране. **Один choke point, никаких точечных проверок у вызывающих.**

### Если единой точки сохранения нет

По результатам `SH1` может выясниться, что сохранение раскидано по нескольким местам. Тогда **до** реализации запрета свести сохранение к одной точке — это минимальный необходимый рефакторинг, он оправдан требованием и должен быть отдельным, изолированным изменением в дифе. Не совмещать его с другими правками этапа. Если объём такого рефакторинга выходит за рамки очевидного — **остановиться и спросить**, а не делать точечные проверки «пока так».

### Что при этом остаётся разрешённым

- Создание **новой** общей настройки из режима `sharedId` (`SH5`, решение 6 в `SH0`) — это запись в общие настройки, а не в личные параметры пользователя. Choke point обязан блокировать именно личные строки (`КодНастройкиОбщей = 0`), а не всякую запись в таблицу. Записи с ненулевым `КодНастройкиОбщей` проходят.
- Переименование и удаление своих общих настроек (`SH7`).

### Признак режима на экране

Пользователь должен понимать, почему его изменения не запоминаются. Добавить неброский индикатор: например чип/подпись рядом с заголовком грида на русском, сообщающая, что открыты общие настройки и изменения не сохраняются. Без модальных окон и без блокировки элементов управления — менять настройки в рамках сессии можно, они просто не переживают перезагрузку.

## 6. Ловушки

1. **Фильтр по пользователю внутри объекта БД.** Объект `UserParamsShared` получает только `@КодНастройкиОбщей` и не должен фильтровать по `КодНастройкиКлиента`: ссылку открывает другой пользователь. На стенде это обеспечено телом функции (`SH2`, п. 5a), но в целевой БД объект чужой — **проверить его поведение под другим пользователем** и, если фильтр по пользователю внутри есть, сообщить заказчику: функционал в этом случае не работоспособен по объективной причине, а не из-за кода приложения.
2. **Сохранение при инициализации.** Некоторые обработчики сохраняют параметры как побочный эффект применения значения. Применение чужого набора в п. 4 не должно вызвать запись в личные параметры — проверить это явно (сравнить содержимое `ClayGridUserParams` до и после открытия по ссылке).
3. **Переход на тот же грид без `sharedId`.** Пользователь ушёл со страницы и вернулся без параметра — режим обязан выключиться, сохранение снова работает. Флаг не должен «залипать» в состоянии компонента или сервиса на весь сеанс.
4. **Быстрый поиск.** Функционал из задачи быстрого поиска сохраняет свой параметр в `ClayGridUserParams` — он обязан подчиняться тому же choke point.

## Чего НЕ делать

- Не предлагать «применить эти настройки себе» — вне рамок.
- Не блокировать элементы управления гридом в режиме `sharedId` (пользователь вправе покрутить настройки в рамках сессии).
- Не показывать сообщение о несоответствии повторно при каждом рендере — один раз на открытие.
- Не писать в БД признак «настройка просмотрена», не вести статистику переходов.

## → verify

1. Unit-тесты на разбор `sharedId`: нет / `0` / пусто / `abc` / `-1` / `7` — по таблице п. 1.
2. Unit-тесты на проверку имён (чистый метод): полное совпадение, подмножество, лишнее имя, **смешанный набор (знакомые имена + одно незнакомое) → несоответствие**, регистр, пустой набор, имя длиннее 20 символов.
2a. Unit-тест-страховка: смешанный набор **не** приводит к частичному применению — метод возвращает несоответствие, а не отфильтрованный список.
3. Открыть ссылку **другим пользователем** (подменив `КодНастройкиКлиента` в БД или зайдя другой учётной записью) → настройки применились, грид загрузился.
3a. Приоритет ограничений: создать ссылку с непустым набором колонок быстрого поиска, затем обнулить `УчаствуетВБыстромПоиске` у этих колонок (`UPDATE ... SET УчаствуетВБыстромПоиске = 0`) → открытие по ссылке даёт грид **без** поля поиска, запрос не содержит условия поиска.
3b. То же с полным удалением колонки `УчаствуетВБыстромПоиске` → грид открывается, поля поиска нет, ошибок и красных баннеров нет.
4. **Ключевая проверка:** снять содержимое `ClayGridUserParams` для своего пользователя, открыть грид по `sharedId`, поменять группировку, фильтры, сортировку, порядок и видимость колонок, быстрый поиск, размер страницы, применить диалог настройки колонок, перезагрузить страницу без `sharedId` → **личные параметры побайтово те же, что были**.
5. Открыть по `sharedId` и нажать «Поделиться» → новая общая настройка создалась, личные параметры не изменились.
6. `sharedId` от другого грида → сообщение об ошибке, **грид не загружен**: таблицы и панели инструментов нет. В профилировщике/логе видно, что запрос данных грида **не выполнялся**.
7. `sharedId` несуществующий → поведение как в п. 6.
8. `sharedId=abc`, `sharedId=-1`, `sharedId=` (пусто) → поведение как в п. 6.
9. `sharedId=0` и отсутствие параметра → обычный режим, никаких сообщений.
10. Кнопка «Открыть без общих настроек» → переход на чистый URL, грид загрузился на личных настройках, сохранение снова работает.
11. Уйти со страницы и вернуться без `sharedId` → режим выключен, изменения снова сохраняются.
12. Индикатор режима виден и понятен; в обычном режиме отсутствует.
13. Прямых запросов к `ClayGridUserParams` по `КодНастройкиОбщей` в новом коде нет (поиск по решению).
14. `dotnet build` + `dotnet test` зелёные.
