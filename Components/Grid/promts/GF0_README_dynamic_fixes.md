# GF — багфиксы динамического режима ClayGrid (оркестратор)

Серия по итогам тестирования G4/G7. Правила работы — те же, что в `_readme_grid_dynamic.md`
(разделы «Как агент работает с КАЖДЫМ промтом» и «Общие правила»). Один файл = один заход =
один коммит. Порядок строгий: он подобран так, чтобы после каждого шага сборка была зелёной,
а блок «Проверка» — выполнимым.

## Симптомы, от которых идём

1. Не применяется настройка колонок по умолчанию — ни порядок, ни видимость.
2. Данных в гриде нет: SQL выполняется, заголовки колонок есть, количество записей и число
   страниц считаются верно, но строк нет и по страницам не листает.

## Диагноз (кратко, подробности — в каждом файле)

| # | Дефект | Симптом |
|---|--------|---------|
| 1 | `InitDynamicMode()` не вызывает первую загрузку данных: заканчивается на `_dynamicInitDone = true`. В статическом режиме стартовую загрузку делает `ClayGridPageBase`, в динамическом — некому | `TotalCount = 0` → `_totalPages = 1` → кнопки пагинатора `Disabled` |
| 2 | `Items = (IEnumerable<TEntity>)rows` — `InvalidCastException`: `TEntity = IClayGridRow`, а `QueryPagedRowsAsync` отдаёт `List<ReadOnlyDictionary<string,object?>>`. В `LoadDynamicData` `TotalCount` присваивается ДО `Items` | счётчик записей верный, строк нет |
| 3 | Разметка ждёт `context.Item is IDetailRow` / `is GroupHeaderRow`, cell-шаблоны ждут `ctx.Item is IReadOnlyDictionary<string,object?>`, а `DetailRow<T>` ограничен `where T : Entity` — общего типа строки нет | пустые ячейки, нет карандаша/корзины |
| 4 | `OnParametersSet()` безусловно делает `_pageNumber = PageNumber` (=1); в динамическом режиме `PageNumber` снаружи не передают | сброс на первую страницу при ре-рендере родителя |
| 5 | `Where(c => c.Order is > 0)` выбрасывает колонки с `Порядок` 0/NULL насовсем — их нет ни в `_columnById`, ни в `_hiddenSqlNames`, ни в диалоге настройки | скрытая по умолчанию колонка недоступна навсегда |
| 6 | `ApplySavedColumns`/`ApplyUrlColumnsValue` чистят `_columnOrder` и заполняют только тем, что в сохранённой строке. `SaveDynamicState()` пишет после КАЖДОЙ загрузки, `CLID` по умолчанию `0` → состояние фиксируется при первом же открытии, до действий пользователя | дефолт из `Порядок` не применяется никогда |
| 7 | `BuildColumnSettingsItems()` сортирует по `_columnOrder.IndexOf(...)`; фильтр-онли колонки (Тип 6/11) есть в `_columnBySqlName`, но не в `_columnOrder` → `IndexOf` = −1 | фильтр-онли всплывают наверх диалога и после «ОК» ломают порядок |

## Волна 1 — заявленные баги (обязательна, порядок строгий)

| Файл | Что чинит | Дефекты |
|------|-----------|---------|
| `GF1_dynamic_row_type.md` | тип строки `ClayDynamicRow` + `LoadDynamicData` оборачивает строки вместо каста | 2, 3 |
| `GF2_initial_load.md` | первая загрузка данных в динамическом режиме | 1 |
| `GF3_paging_reset.md` | не сбрасывать `_pageNumber` в динамическом режиме | 4 |
| `GF4_default_hidden_columns.md` | скрытая по умолчанию = зарегистрирована, а не выброшена | 5 |
| `GF5_columns_state_merge.md` | сохранённое состояние колонок мерджится с дефолтом | 6 |
| `GF6_settings_dialog_order.md` | источник порядка в диалоге — только `_columnOrder` | 7 |

**GF1 строго перед GF2**: без него первая загрузка просто уронит грид `InvalidCastException`.
**GF4 строго перед GF5**: мердж опирается на то, что скрытые колонки зарегистрированы.
GF3 и GF6 внутри волны независимы, но GF3 осмысленно проверять после GF2, GF6 — после GF4.

## Волна 2 — находки из разбора (`GF7_backlog.md`)

Не блокируют волну 1, но каждая — живой дефект. Порядок — по возрастанию цены.

| Файл | Что чинит | Зависит от |

| `GF9_filter_clear_persist.md` | снятый фильтр возвращается после F5 | — |

| `GF14_disable_dynamic_grouping.md` | группировка: UI есть, результата нет, колонка ещё и прячется | GF1, GF2 |
| `GF15_disable_dynamic_export.md` | печать и Excel: меню есть, клики молча ничего не делают | GF1, GF2 |
| `GF16_search_parameter.md` | поиск падает `SqlException`: `BuildWhereClause` генерирует `LIKE @search`, а `dp.Add("search", ...)` в `LoadDynamicData` нет | GF1, GF2 |

GF14 и GF15 — **убирают нерабочий UI, а не реализуют фичи**. Группировки и экспорта в
динамическом режиме никогда не было: `LoadDynamicData` игнорирует `GroupColumns`, а все точки
входа печати/Excel начинаются с `if (DataLoader is null) return;`. Реализация — отдельный план,
см. `GF7_backlog.md`, пункты 1–2. Если заказчику эти функции нужны сейчас — останавливайся
на GF13 и планируй фичу, а не выполняй GF14/GF15.

## Волна 3 — фичи (не багфиксы)

`GG0_README_dynamic_grouping.md` + `GG1`–`GG8` — реализация группировки динамического грида.
GG7 частично откатывает GF14. Начинать только после волны 1 и после GF16.

## Что остаётся за серией

`GF7_backlog.md` — что промтом не закрывается: группировка и экспорт динамического грида,
фильтр по значению, неиспользуемые `Пиктограмма` и `IDName`, модель доверия к SQL из
справочника. Не бери в работу без отдельного указания.
