# GB — багфиксы по итогам тестирования динамического грида и диалога колонок (оркестратор)

Серия по итогам ручного тестирования после волн GF / GG / GN / GE. Правила работы — те же, что
в `_readme_grid_dynamic.md` (разделы «Как агент работает с КАЖДЫМ промтом» и «Общие правила»)
и в корневом `/AGENTS.md` (Think Before Coding / Simplicity First / Surgical Changes /
Goal-Driven Execution).

**Один файл = один заход = один коммит. Не забегай вперёд. Не делай два файла за раз.**
Коммит — только по прямому указанию (см. `/AGENTS.md`, Agent instructions).

Куда положить эти файлы в репозитории: `src/Clayzor.Lib.Web.Controls/Components/Grid/promts/`.

## Симптомы, от которых идём (со слов тестировщика)

1. Невозможно выбрать записи для групповых операций в динамическом гриде — кнопки нет на панели.
2. При экспорте в Excel не подгружаются данные закрытых групп.
3. При операциях экспорта не видно, что идёт долгая операция.
4. Диалог настройки колонок кривой: при изменении ширины кнопки наезжают друг на друга;
   названия колонок в шапке не совпадают по горизонтали с самими колонками; при прокрутке
   списка чипы видны в щели между шапкой таблицы и заголовком диалога.

## Волна 3 (баги из второго прогона тестирования)

Четыре независимых бага, найденных при тестировании динамического грида:

1. **Нет режима ожидания** при группировке/сортировке/фильтрации в динамике → `GB12`.
   Заказчик: пока отдельного индикатора нет, использовать тот же оверлей, что в групповых
   операциях (`.clay-grid-busy` из GB3). Поэтому GB12 идёт **после GB3**, чтобы переиспользовать
   его `RunBusyAsync`, а не плодить второй индикатор.
2. **Прыжок прокрутки** при раскрытии группы (`@key="_dataKey"` пересоздаёт грид) → `GB13`.
   Независим, порядок любой.
3. **Value-filter не включается** в динамике (`AllowValueFilter` не задаётся при построении
   `ClayColumnMeta`) → `GB14`. Независим.
4. **Список колонок в фильтре не по алфавиту** → `GB15`. Независим, одна строка правки.

GB13/GB14/GB15 друг от друга и от остальной серии не зависят — можно делать в любой момент.
GB12 привязан к GB3 (общий индикатор) и к GB8 (сохранение/загрузка из обработчиков, MARS).

## Решения заказчика (зафиксированы, не переспрашивать)

1. **Свёрнутая группа в Excel** — строка группы одна, свёрнутая, но подчинённые данные в файле
   ЕСТЬ; пользователь разворачивает узел средствами Excel (Outline). Именно так и сделано в
   GB2/GB6: `expandedGroups` по-прежнему уезжает в `ClayGridExcelGenerator`.
2. **Дубли в статическом экспорте** — чинить сейчас, отдельным шагом (`GB6`), а не в бэклог.
3. **Стили должны быть общие** — дублирование `app.css` устраняется в рамках серии (`GB7`),
   общий файл живёт в RCL.
4. **Печать «Текущая страница»** — строго то, что на экране: свёрнутая группа печатается одной
   строкой-заголовком, раскрытая группа, разорванная пагинацией, печатается только видимой
   частью (догрузки хвоста нет). См. `GB10`.
5. **Печать «Все данные»** — весь список, все группы раскрыты; состояние свёрнутости на экране
   игнорируется. Это единственное исключение из правила п. 4. См. `GB10`.

## Общее для всех файлов серии

- **CSS.** До выполнения GB7 `.column-settings-*`, `.clay-column-settings-*`, `.clay-print-spinner`
  и прочие классы грида лежат в ДВУХ файлах: `src/Clayzor.App.Web.MedicalTests/wwwroot/css/app.css`
  и `src/Kesco.App.Web.Inventory/wwwroot/css/app.css`, и копии уже разъехались
  (`margin-bottom: -4px` против `2px`, у Kesco есть `.chip-group-badge` и `.grouping-tray-add-btn`,
  у MedicalTests нет). Пока GB7 не сделан — любую правку CSS вносить в ОБА файла дословно
  одинаково. После GB7 общий стиль правится только в
  `src/Clayzor.Lib.Web.Controls/wwwroot/css/clay.css`.
- **StyleGuard.** `build/StyleGuard.targets` роняет сборку при визуальных инлайнах в `.razor`/`.cs`
  (`color`, `background*`, `border*`, `font-*`, hex, `rgba(`). Структурные инлайны
  (`display`, `flex`, `gap`, `width`, `padding`, `position`, `z-index`, `overflow`) разрешены.
  Новый визуальный паттерн → класс в `app.css` + строка в таблицу `STYLE_RULES.md` §4.
- **Проверочный стенд.** Динамический грид — `Kesco.App.Web.Inventory`, страница `/?id=<КодЗапроса>`
  (в промтах серии GF/GG использовался `?id=140`). Статический — `Clayzor.App.Web.MedicalTests`,
  `/medical-tests`. **Каждый фикс проверяется в ОБОИХ режимах**: общий код грида один.
- После каждого файла: `dotnet build Clayzor.sln` + `dotnet test tests\Clayzor.Lib.Web.Controls.Tests`
  + блок «Проверка» из файла.

## Не входит в серию

- Реализация выбора строк для гридов с нечисловым ID (`_selectedIds` — `HashSet<int>`,
  см. «Не делай» в GB1).
- **Соединение на каждый запрос вместо одного на circuit** (`DbManager`). По существу — правильная
  модель для веб-приложения, снимающая проблему GB8 в корне, но это смена контракта DALC.
  Кандидат в бэклог, решение за заказчиком. GB8 закрывает симптом сериализацией — надёжно,
  но это не отменяет вопроса.
- **Загрузка ID потомков групп из `OnAfterRenderAsync`** (GG8). После GB8 не падает, но ходить
  в БД из рендера — плохое место; правильное — путь загрузки данных и `ToggleSelectMode`.
  Кандидат в бэклог.
- Сведение `ClayGridPageBase.Export.*` и `ClayGrid.Dynamic.Export.cs` к общему коду: после
  GB2/GB6 методы становятся близнецами с тремя отличиями (источник строк, обёртка строки,
  чтение ключа). Кандидат в отдельный рефакторинг — но не в багфикс.
- Переход на CSS isolation, дробление `clay.css`, чистка legacy-алиасов `--lh-*` (см. «Не делай»
  в GB7).
- **Сохранение `_valueFilterDisabledColumns`** (переключатель «Фильтр по значению» в диалоге
  настройки колонок): не сериализуется вообще, теряется после F5. Нужен новый параметр в
  `vwНастройки` и решение заказчика по имени префикса. Находка GB11.
- Фильтр по значению в динамическом режиме (`GF7_backlog.md`, п. 3) и остальные пункты бэклога.
